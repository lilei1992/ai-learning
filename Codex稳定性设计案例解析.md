## 写在前面

如果说 Claude Code 是"用 Claude 模型的 Agent 客户端"，那 Codex（OpenAI 推出的命令行编程 Agent）就是"用 GPT 系列模型的 Agent 客户端"。

两者的**外壳长得不一样**（一个调用 Claude API，一个调用 OpenAI Responses/Chat API），但**底层要解决的工程问题是同一套**：

> 模型会超时、API 会限流、网络会抖动、一个任务要循环调用很多次模型、每次调用都要花钱。

这篇文档用 Codex 的真实使用场景，再过一遍这六个概念，帮助你从"两个不同产品"中**抽象出同一套设计模式**——这才是真正值钱的东西。

## 一、超时 Timeout —— "这个 shell 命令我等不起"

### 场景

你在 Codex 里说："帮我跑一下 `pytest` 看看测试结果"。

如果这个项目的测试套件里有一个测试**死循环了**（比如 `while True` 忘了写退出条件），命令会一直占用进程，永远不返回。

### Codex 的做法

Codex 在执行 shell 命令时，**每个命令都有自己的执行时长上限**。一旦命令运行超过这个时间：

```
执行: pytest tests/
   ↓
运行中... 30s... 60s... 120s...
   ↓ 达到超时阈值
强制终止进程，回收资源
   ↓
返回给模型: "命令执行超时（timeout），已被终止"
   ↓
模型看到这个结果后会调整策略，比如：
"看起来某个测试卡住了，我先用 -k 参数只跑部分测试用例排查"
```

### 用大白话理解

想象你点了一份"现烤披萨"外卖，正常 30 分钟送到。如果商家系统设定"超过 90 分钟未送达自动取消订单并退款"，那这个**90 分钟就是超时阈值**——它的意义不在于"惩罚商家"，而是**让你这个等待方有一个确定的结果**，不用一直焦虑地刷新订单页面。

### 对应到 Java 后端

```java
// MoreTickets场景：调用AI生成的"日志分析摘要"
// 长日志可能导致模型生成时间很长，必须设置硬超时

CompletableFuture<String> future = CompletableFuture.supplyAsync(() ->
    aiClient.analyzeLog(longLogContent)
);

try {
    String result = future.get(60, TimeUnit.SECONDS); // 硬超时60秒
} catch (TimeoutException e) {
    future.cancel(true); // 主动取消，释放连接资源
    // 走降级：只分析日志的前1000行
    return aiClient.analyzeLog(truncate(longLogContent, 1000));
}
```

## 二、重试 Retry —— "服务器说'我现在忙，等一下'，那就等一下"

### 场景

Codex 跑批量任务时（比如让它"扫描整个项目里所有 TODO 注释，逐一分析"），会**连续多次调用模型**。如果你的账号短时间内调用次数太多，OpenAI API 会返回：

```
HTTP 429 Too Many Requests
{
  "error": {
    "message": "Rate limit reached for requests",
    "type": "rate_limit_exceeded"
  }
}
```

### Codex 的做法

收到 429 后，Codex（以及 OpenAI 官方 SDK）的标准做法是：

1. **先看响应头里有没有 `Retry-After` 或限流相关的提示**——服务器明确告诉你"建议等 X 秒再来"，那就照做。
2. **没有明确提示的话，用指数退避**：第 1 次等 1 秒，第 2 次等 2 秒，第 3 次等 4 秒……
3. **加上随机抖动（Jitter）**：避免"同一秒被限流的所有请求，又在同一秒一起重试"，导致二次拥堵。

```
请求1: 429 Too Many Requests
   ↓ 等待 1秒 + 随机(0~0.5秒)
请求2(重试): 429 Too Many Requests（还在限流窗口内）
   ↓ 等待 2秒 + 随机(0~1秒)
请求3(重试): 200 OK ✅ 限流窗口过去了，成功
```

### 用大白话理解"为什么不能无脑重试"

把"调用 AI API"想象成"打客服电话"：

- 客服说"线路繁忙，请稍后再拨"——你**立刻**狂按重拨键，结果就是**你和所有跟你一样心急的人**，把客服线路堵得更死。
- 正确做法是：**先去倒杯水（等待），过一会儿再拨**，而且最好**不同的人错开时间**（jitter），这样客服才有机会消化掉积压的来电。

### 哪些情况"重试也没用"——Codex 不会做的事

```
场景：你的Prompt里粘贴了一份50万字的文档，超过了模型最大输入长度

请求: 400 Bad Request - "context_length_exceeded"

Codex的反应: ❌ 不会重试（重试10次结果都一样）
            ✅ 会调整策略：把文档分段，或者只截取关键部分重新发送
```

```
场景：你的API Key过期了

请求: 401 Unauthorized

Codex的反应: ❌ 不会重试
            ✅ 直接提示用户："认证失败，请检查API Key配置"
```

## 三、限流 Rate Limiting —— "OpenAI 的 TPM/RPM 限制，就是悬在头上的天花板"

### 场景

OpenAI 给每个账号设置了两种限制（这是 Codex 这类工具必须遵守的"游戏规则"）：

- **RPM**（Requests Per Minute）：每分钟最多发起多少次请求
- **TPM**（Tokens Per Minute）：每分钟最多消耗多少 Token

```
假设你的账号限制：
  RPM = 500  （每分钟最多500次请求）
  TPM = 200,000  （每分钟最多20万Token）

场景：Codex帮你"批量重构100个文件"，
每个文件分析+生成大约消耗3000 Token，调用1次

如果Codex不限流，直接100个请求一起发出去：
100次请求 × 3000token = 30万Token > 20万TPM上限
   → 后面的请求会陆续收到429，体验很差 ❌

如果Codex内部做了节流（控制并发+间隔）：
分批发送，比如每分钟发60个请求（60×3000=18万 < 20万TPM）
   → 平稳完成，不触发429 ✅
```

### 用大白话理解：水龙头和水管

你家的自来水管**最大流量是固定的**（比如每分钟100升）。如果你同时打开厨房、卫生间、洗衣机三个水龙头，**总流量不会超过水管上限**——只是每个龙头分到的水变少了，或者水压下降。

OpenAI/Anthropic 的 TPM/RPM 限制就是这根"主水管"——**不管你客户端怎么设计，这个上限是硬约束**。所以客户端（Codex / 你的Java服务）能做的就是：**自己先排队控流，别一下子把水龙头全开到最大**，这样才能保证整体平稳，而不是"水压瞬间归零"。

### 对应到你的业务

```java
// MoreTickets场景：营销活动期间，批量为1000个商品生成AI文案

// ❌ 错误做法：一次性提交1000个异步任务
for (Product p : products) {
    executorService.submit(() -> aiService.generateCopy(p)); // 瞬间打满TPM限制
}

// ✅ 正确做法：用限流器控制提交速率
RateLimiter limiter = RateLimiter.create(5.0); // 每秒最多5个请求
for (Product p : products) {
    limiter.acquire(); // 阻塞直到获得许可
    executorService.submit(() -> aiService.generateCopy(p));
}
```

## 四、隔离 Bulkhead —— "重构100个文件 vs 回答一个简单问题，不该抢同一份资源"

### 场景

假设你正在用 Codex 跑一个"批量重构"的大任务（要处理几十个文件，每个文件都要调用模型分析+生成代码）。这个任务会持续运行很久。

同时，你**临时打断它**，问了一句"刚才那个 `utils.js` 里的函数是干什么的？"——这是一个**轻量、快速**的问题。

### 为什么需要隔离

如果"批量重构"和"临时提问"**共享同一个资源池**（比如同一个网络连接队列、同一个并发槓位），那么：

```
没有隔离：
批量重构任务占满了所有"调用槓位"
   ↓
你的临时提问 → 排在队尾，要等批量任务的某个槓位释放才能执行
   → 你问一个简单问题，却要等很久才有反应 ❌
```

```
有隔离：
"交互式提问"和"后台批量任务"分别拥有独立的资源槓位
   ↓
即使批量重构占满了它自己的槓位，你的临时提问依然能立刻被处理 ✅
```

### 用大白话理解：医院的"急诊"和"普通门诊"

医院把**急诊科**和**普通门诊**物理上分开，挂号、候诊、医生资源都是**独立配置**的。

如果不这样设计，普通门诊排了500个人挂号，急诊病人来了也要排在第501位——这显然是不合理的。**隔离**就是给不同优先级、不同性质的请求，**划分专属的"资源车道"**，互不挤占。

### 对应到你的业务

```java
// MoreTickets场景：
// "实时客服AI回复"（用户在线等待，要快） 
// vs "夜间批量生成商品SEO文案"（后台任务，可以慢）

@Configuration
public class AiThreadPoolConfig {

    // 实时交互专用线程池：少而精，保证响应速度
    @Bean("realtimeChatExecutor")
    public ExecutorService realtimeChatExecutor() {
        return new ThreadPoolExecutor(
            5, 5,  // 固定5个线程，专门服务实时聊天
            0L, TimeUnit.MILLISECONDS,
            new ArrayBlockingQueue<>(20) // 排队上限20
        );
    }

    // 批量任务专用线程池：可以多线程但优先级低
    @Bean("batchTaskExecutor")
    public ExecutorService batchTaskExecutor() {
        return new ThreadPoolExecutor(
            10, 10,
            0L, TimeUnit.MILLISECONDS,
            new ArrayBlockingQueue<>(1000) // 排队上限1000
        );
    }
}

// 即使batchTaskExecutor里堆积了999个任务在排队
// realtimeChatExecutor依然有5个空闲线程随时响应用户聊天
```

## 五、熔断 Circuit Breaker —— "OpenAI 服务大规模故障时，别让用户陪你一起卡死"

### 场景

OpenAI/Azure OpenAI 偶尔会出现**区域性服务故障**（官方状态页会标红）。如果这种时候 Codex 还是"每次请求都老老实实等到超时才报错"，那用户体验会是：

```
你: "帮我看看这段代码"
[等待5秒... 10秒... 30秒... 60秒]
Codex: "抱歉，请求超时了"

你: "好吧，再试一次"
[又等待60秒]
Codex: "抱歉，又超时了"

...用户已经崩溃 😫
```

### 熔断的核心思想（再用一个新类比）

> 想象你常去的一家奶茶店，最近这家店"评分断崖式下跌"——最近20个顾客的评价里，12个都是"等了40分钟没拿到"。

这时候，**美团/饿了么平台**应该怎么做？

```
🟢 正常状态（CLOSED）：
   平台正常推送这家店给用户，正常下单

🔴 熔断状态（OPEN）：
   平台监测到"最近20单超时率60% > 阈值50%"
   → 临时把这家店标记为"暂不可下单"
   → 用户搜索时直接看不到这家店（或显示"暂停营业"）
   → 接下来15分钟，所有想点这家店的订单
     直接被引导去"附近其他奶茶店"（降级）
   → 不会让任何用户再去"踩坑"，体验更可控

🟡 半开探测（HALF_OPEN，15分钟后）：
   平台派"测试订单"去看这家店恢复了没
   → 如果这几个测试订单都正常出餐 → 重新开放（回到CLOSED）
   → 如果还是慢 → 继续熔断，再等15分钟
```

### 对应 Codex / AI 调用场景

```
最近20次调用GPT-4模型的统计：
  成功: 8次
  超时/5xx错误: 12次
  → 错误率60% > 阈值50%
  → 熔断器跳转到OPEN

接下来60秒内：
  所有调用GPT-4的请求 → 不再尝试（避免浪费60秒等超时）
  → 直接切换到备用模型（比如GPT-4o-mini，或者本地缓存的常见回答）

60秒后，HALF_OPEN：
  放行5个探测请求去GPT-4
  → 如果都成功，恢复正常（CLOSED）
  → 如果还失败，继续熔断（回到OPEN，再等60秒）
```

### 一句话理解熔断和重试的区别

> **重试**解决的是"这一次请求要不要再来一遍"；
> **熔断**解决的是"接下来一段时间，这条路要不要彻底别走了"。

重试是"微观、单次"的决策；熔断是"宏观、一段时间"的决策。

## 六、降级 Fallback —— "Codex 没法联网搜索时，它会怎么办？"

### 场景

假设你让 Codex"帮我查一下某个最新发布的 npm 包的用法"。如果 Codex 当前**没有网络访问权限**，或者网络请求失败了，它不会直接说"我不知道，结束对话"。

### Codex（以及一般 Agent）的降级思路

```
Level 0（最理想）：
  联网搜索 → 获取该npm包最新文档 → 给出准确答案
   ↓ 网络不可用 / 搜索失败

Level 1（降级1）：
  基于训练数据中的已知信息回答
  → 但会明确告知："我的知识可能不是最新的，
     建议你查看官方文档确认最新用法"
   ↓ 如果这个包训练数据里完全没有

Level 2（降级2）：
  根据包名的命名规律和类似包的通用模式，给出"大概率正确"的猜测用法
  → 明确标注"这是推测，请自行验证"
   ↓

Level 3（兜底）：
  诚实告知："我对这个包不熟悉，建议你运行
  `npm view <package> readme` 来查看包自带的说明文档"
  → 给用户一个"自己动手"的明确路径，而不是干巴巴的"我不知道"
```

### 用大白话理解："降级"不是"敷衍"，而是"管理期望"

很多人以为"降级"就是"糊弄用户"。其实恰恰相反——**好的降级是在诚实地告诉用户"我现在能给到的最好结果是什么程度"**，而不是让用户**长时间等待，最后得到一个"系统错误"的冰冷提示**。

类比：你去理发店，理发师说"今天这个发型师没来，要不要让我这个总监给你剪？价格一样但风格可能略有不同"——这就是**清晰的降级沟通**，比"对不起，今天不能为您服务"体验好得多。

### 对应到你的业务

```
场景：MoreTickets「AI智能客服」回复用户咨询

Level 0: GPT-4o + 完整订单上下文 + RAG检索知识库
   → 精准回答："您的订单#12345已发货，预计明天下午到达"
   ↓ RAG检索服务超时

Level 1: GPT-4o + 完整订单上下文（不检索知识库，靠模型自身能力）
   → 较准确回答，但可能缺少最新政策信息
   ↓ GPT-4o熔断中

Level 2: GPT-4o-mini + 订单基础信息（不含完整上下文）
   → 简化回答："您的订单正在处理中，具体物流信息建议查看订单详情页"
   ↓ 所有AI服务都不可用

Level 3: 规则模板匹配
   → 根据关键词"发货""物流"匹配预设话术：
     "您可以在【我的订单】页面查看实时物流状态"
   ↓

Level 4: 友好兜底 + 转人工
   → "抱歉，智能客服暂时繁忙，已为您转接人工客服，
      预计等待时间3分钟"
```

## 七、综合案例：用 Codex 完成"修复一个 Bug 并提交 PR"

把六个概念串联起来看一个完整任务：

> "帮我修复 issue #234 描述的 bug，写好测试，然后提交一个 PR"

```
1.【限流】
   Codex开始任务前，先评估这个任务大概需要调用模型多少次
   （读代码、分析、改代码、跑测试、写PR描述...）
   → 自身内部会控制调用节奏，避免短时间内打爆TPM/RPM

2.【超时 + 隔离】
   第一步：读取issue描述、相关源码文件
   → 每个文件读取都有独立超时（几秒级别）
   → 文件读取这种轻量操作，和后面"跑测试"这种重操作
      使用不同的资源池，互不影响

3.【正常调用，偶发抖动 → 重试】
   分析代码，调用GPT-4进行推理
   → 假设第一次调用遇到网络瞬断 → 自动重试（指数退避+jitter）
   → 第二次成功，拿到分析结果

4.【超时】
   修改代码后，运行测试：`npm test`
   → 设置较长但有限的超时（比如180秒）
   → 假设某个测试因为改动引入了死循环
   → 超时后强制终止，告知模型"测试卡住了"

5.【降级】（基于上面的超时信息）
   模型收到"测试超时"的反馈后
   → 降级策略：先检查是不是刚才的修改引入了循环依赖
   → 如果定位到问题，修复后重新跑测试

6.【熔断】（假设极端场景：OpenAI API大范围故障）
   如果在任务执行过程中，连续多次调用都失败（错误率超阈值）
   → 熔断打开，Codex会暂停当前任务
   → 提示用户："当前AI服务不稳定，任务已暂停，
      你可以稍后输入'继续'恢复任务"
   → 而不是让你眼睁睁看着每一步都转30秒圈最后报错

7.【最终交付】
   测试通过 → 生成PR描述 → 提交
   → 全程对用户来说是"流畅的一次性体验"
   → 但背后是六层稳定性机制在保驾护航
```

## 八、Claude Code vs Codex：本质是同一套"免疫系统"

| 维度 | Claude Code（调用Claude模型） | Codex（调用GPT模型） | 本质 |
|------|------------------------------|---------------------|------|
| 超时 | Bash工具超时强制终止 | Shell命令超时强制终止 | 给慢请求一个截止时间 |
| 重试 | 529/网络抖动 → 指数退避重试 | 429/5xx → 指数退避重试 | 临时故障自愈，错开重试时间 |
| 限流 | 遵守Anthropic的速率限制 | 遵守OpenAI的TPM/RPM限制 | 不打爆上游，平稳消费配额 |
| 隔离 | 不同工具调用资源独立 | 交互式任务 vs 批量任务资源独立 | 一个慢操作不拖累其他操作 |
| 熔断 | （理念上）持续失败时暂停尝试 | （理念上）持续失败时暂停尝试 | 故障期间不做无谓等待 |
| 降级 | 模型不可用时切换备用/简化策略 | 网络/工具不可用时给出替代建议 | 永远给用户一个"可接受的结果" |

**核心结论**：

> 不管是 Claude Code、Codex，还是你正在为 MoreTickets 设计的 AI 推理服务——**只要你的系统依赖一个"不是 100% 可靠、有调用成本、响应时间不固定"的外部服务**，这套"超时-重试-限流-隔离-熔断-降级"的组合拳就是**标准配置**，不是"高级技巧"，而是**及格线**。

---

## 九、给新手的实践建议

如果你是刚接触这套理论的 Java 后端，建议按这个顺序练手：

```
第1步：先加超时
  给你现有的AI调用加上明确的连接超时和读取超时
  （这是最简单、收益最大的第一步）

第2步：加重试 + 区分异常类型
  写一个RetryTemplate，区分"可重试"和"不可重试"异常
  （记住：429和5xx重试，400和401不重试）

第3步：加限流
  用Guava RateLimiter做单机限流，控制调用频率
  （先单机，再考虑Redis分布式）

第4步：加熔断
  引入Resilience4j，给每个模型配置独立的CircuitBreaker
  （观察一段时间的错误率，再调整阈值）

第5步：设计降级链
  至少准备2层降级：备用模型 + 友好提示
  （先有"兜底"，再逐步丰富中间层级）

第6步：加隔离
  如果有"实时"和"批量"两类场景，给它们分配独立线程池
  （避免互相影响）

最后：补监控
  把以上每一步触发的次数都打成指标，上报到Prometheus/Grafana
  （没有监控，你永远不知道这些机制有没有真正起作用）
```
