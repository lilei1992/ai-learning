# 从 Claude Code 看 AI 推理稳定性设计：新手生动指南

> 用一个你每天都在用的工具，理解限流/降级/熔断/超时/重试/隔离到底是什么

---

## 写在前面

Claude Code 本质上也是一个"AI 推理调用客户端"——它每次帮你改代码、读文件、跑命令，背后都是在调用 Claude 模型 API。

它面临的问题，和你做 MoreTickets 的 AI 客服回复、商品文案生成是**完全一样的**：

- 模型有时候慢，有时候报错
- 一次对话可能消耗几万 Token，很贵
- 上游 API 会限流（429）
- 网络会抖动
- 一个任务可能需要调很多次模型（多轮 Agent 循环）

下面我们用"Claude Code 在实际使用中会遇到的真实场景"，把每个稳定性概念讲清楚。

---

## 一、超时 Timeout —— "这个命令卡住了，要不要等？"

### 场景

你让 Claude Code 执行 `npm install`，这个命令可能 5 秒完成，也可能因为网络问题挂起 10 分钟。

### Claude Code 的做法

Claude Code 在执行 Bash 命令时会有**默认超时时间**（通常是 2 分钟），超过就会自动终止命令，并把"超时"这个信息返回给模型，让模型决定下一步怎么办（比如换一种方式重试，或者提示用户网络可能有问题）。

```
工具调用: bash_tool("npm install")
   ↓
等待 120s...
   ↓
超时！强制 kill 进程
   ↓
返回给模型: "Command timed out after 120000ms"
   ↓
模型决定: "看起来网络较慢，我用 --registry 切换镜像源重试"
```

### 对应到你的业务

| Claude Code | 你的 AI 客服系统 |
|---|---|
| Bash 命令超时 → kill 进程 + 告知模型 | 模型调用超时 → 断开连接 + 走降级 |
| 不同命令超时时间不同（npm install 比 ls 久） | 不同业务超时不同（"总结长文档"比"生成短文案"久） |

### 一句话理解

> **超时不是"等到天荒地老"，而是"给一个明确的截止时间，超时就主动放手"**——否则一个卡住的请求会一直占着资源不放，最后整个系统都会被这些"僵尸请求"堵死。

## 二、重试 Retry —— "刚才网络抖了一下，再试一次"

### 场景

Claude Code 调用 Anthropic API 时，偶尔会遇到网络波动，返回 `529 Overloaded`（服务器过载）或连接被重置。

### Claude Code 的做法

Claude Code 的底层 SDK（Anthropic SDK）内置了**自动重试机制**：

```
第1次调用 → 失败（529 Overloaded）
   ↓ 等待 ~1秒（带随机抖动）
第2次调用 → 失败（连接超时）
   ↓ 等待 ~2秒（带随机抖动）
第3次调用 → 成功 ✅
```

关键点：

1. **只对"临时性错误"重试**——529（过载）、连接超时、网络中断这些"网络抖了一下"的情况会重试。
2. **不对"确定性错误"重试**——比如你的 API Key 错了（401），或者你的 Prompt 格式不对（400），重试 100 次结果都一样，纯粹浪费钱和时间。
3. **每次重试都要等一会儿，而且间隔越来越长**（指数退避），避免"我重试 → 服务器更忙 → 我再重试 → 服务器彻底崩溃"的恶性循环。

### 用大白话理解为什么不能"立即重试"

想象一个奶茶店，排队的人太多店员处理不过来。这时候如果**所有排队的人立刻又插队重新排一次**，店里只会更乱。正确做法是：大家**错开时间**再来，店员才有机会喘口气恢复正常。

这就是为什么需要：
- **指数退避**：第一次等 1 秒，第二次等 2 秒，第三次等 4 秒……给系统恢复的时间
- **Jitter（随机抖动）**：如果 1000 个用户都在第 1 秒同时重试，等于又来了一波"集中冲击"。加上随机抖动后，重试请求会分散在 0.5~1.5 秒之间，错开了高峰。

### 对应到你的业务

```java
// MoreTickets 商品文案生成场景
// 用户提交文案生成请求 → AI调用偶发502 → 自动重试

第1次: 调用GPT生成文案 → 502 Bad Gateway
等待 1秒（+随机0~0.5秒）
第2次: 调用GPT生成文案 → 成功，返回文案 ✅
```

但如果是：

```java
第1次: 调用GPT生成文案 → 400 "您的Prompt超过了最大长度"
直接报错，不重试 ❌
（因为重试100次，Prompt还是太长，结果一样）
```

## 三、限流 Rate Limiting —— "排队系统，别一窝蜂涌进来"

### 场景

如果你在 Claude Code 里疯狂连续发消息（比如写脚本批量调用），很快就会撞到 Anthropic API 的速率限制：

```
Error: 429 Too Many Requests
"Rate limit exceeded. Please wait before making another request."
```

### Claude Code 的做法

1. **客户端侧**：Claude Code 一次只会有一个"主任务"在跑，不会同时发出 100 个并发请求打爆你的账号配额。
2. **遇到 429 后**：会读取响应头里的 `retry-after`（服务器告诉你"等 X 秒再来"），然后乖乖等待。
3. **Claude Code 内部的 Agent 循环**（比如连续调用多个工具）也是**串行执行**的，不会一次性把所有工具调用都发出去——这本身就是一种"天然限流"。

### 用大白话理解限流

限流就像**银行柜台的排队系统**：

- 不是"谁先到就先服务"造成的混乱（大家都挤在窗口前）
- 而是**取号排队**：每秒钟只放一个人进去办理，其他人乖乖在外面等，避免大堂拥堵到连消防通道都堵死

```
                  ┌──────────────────┐
  100个请求同时到达 →│   限流闸门         │ → 每秒只放5个进去
                  │  (令牌桶/排队)     │
                  └──────────────────┘
                          ↓
                  AI模型服务（处理能力有限）
```

### 对应到你的业务

```
场景：双11大促，1万个用户同时点击"AI生成专属优惠文案"

没有限流：
1万个请求 → 全部冲向OpenAI API → 上游直接返回429 → 全部失败 ❌

有限流（按租户 + 按用户）：
1万个请求 → Redis限流闸门
  → 每个用户最多1 req/s（防止单用户刷接口）
  → 整个租户最多50 req/s（防止打爆上游配额）
  → 超出的请求进入队列，或友好提示"AI正忙，请稍候"
```

## 四、隔离 Bulkhead —— "Bash 工具卡住了，不影响我读文件"

### 场景

假设你让 Claude Code 同时做两件事：① 跑一个耗时很长的测试命令 ② 读取一个配置文件。

### Claude Code 的做法

Claude Code 中**不同类型的工具调用是相互独立的**：Bash 命令、文件读写、网络请求，各自有自己的执行和超时逻辑。即使 Bash 命令因为某个命令挂起很久，也**不会影响到文件读取这种轻量操作**的正常返回。

这就像轮船的"水密舱"设计——船体被分成多个独立密封舱，**一个舱漏水，不会导致整条船沉没**，因为其他舱是隔离的。

### 对应到你的业务

```
场景：MoreTickets同时有两类AI请求
  ① "生成长篇商品详情页文案"（耗时、Token多、用GPT-4o）
  ② "判断用户咨询是否属于售后问题"（耗时短、Token少、用GPT-4o-mini）

没有隔离：
所有请求共用一个线程池 → 一堆"长文案生成"请求把线程池占满
  → "售后问题分类"这种简单快速的请求也被堵住，用户等很久 ❌

有隔离（按场景分配独立资源池）：
"长文案生成"线程池（10个线程）  ← 互不影响 →  "售后分类"线程池（20个线程）
即使长文案生成全部卡住，售后分类依然秒级响应 ✅
```

```java
// Java实现思路：为不同业务场景配置独立线程池
@Bean("longContentExecutor")
public ExecutorService longContentExecutor() {
    return new ThreadPoolExecutor(10, 10, ...); // 长文案生成专用
}

@Bean("quickClassifyExecutor")
public ExecutorService quickClassifyExecutor() {
    return new ThreadPoolExecutor(20, 20, ...); // 快速分类专用
}
```

## 五、熔断 Circuit Breaker —— "这条路一直堵车，先别走这条路了"

### 场景

假设 Anthropic API 突然大面积故障（官方挂了），Claude Code 如果还是**每次都傻傻地去尝试连接、超时、报错**，体验会非常差——每次操作你都要等上 1-2 分钟才收到一个错误。

### 熔断的核心思想（Claude Code 视角的类比）

虽然 Claude Code 本身没有公开"熔断器"的具体实现细节，但这个思想在任何依赖外部服务的系统里都该有，我们用一个**生动的类比**来理解：

> 想象你每天开车上班要经过一座桥。如果你连续 5 天都发现这座桥"封路施工"，第 6 天你还要不要傻乎乎地开车过去，到了桥头才发现还是封的，再掉头走另一条路浪费 20 分钟？

**正常人的做法**：连续遇到几次封路后，你会**记住这条路"暂时不通"**，接下来一段时间**直接走备用路线**，过一阵子再派人去探探路看是否恢复了。

这就是熔断器的三个状态：

```
🟢 CLOSED（关闭/正常）
   "这条路能走，正常通行"
   每次过桥都正常通过，但你心里默默记录：最近10次有几次是封路的

🔴 OPEN（打开/熔断）
   "最近10次有6次都封路了（错误率60% > 阈值50%），
    接下来60秒我直接走备用路线，不浪费时间去试这条桥了"
   所有请求直接走Fallback（降级），根本不发往AI服务

🟡 HALF_OPEN（半开/试探）
   "60秒过去了，派一个'侦察兵'去看看桥修好了没"
   放行少量请求去真实调用AI服务
   - 如果侦察兵成功回来 → 恢复 CLOSED，正常通行
   - 如果侦察兵又被堵 → 回到 OPEN，再等60秒
```

### 对应到你的业务

```java
场景：MoreTickets调用GPT-4o生成商品文案

正常情况（CLOSED）：
请求1: 成功 ✅
请求2: 成功 ✅
请求3: 失败 ❌ (502)
...
请求20: 统计最近20次请求，错误率55% > 阈值50%
→ 熔断器跳转到 OPEN 状态！

熔断状态（OPEN）：
请求21~请求N（接下来60秒内的所有请求）：
→ 不再尝试调用GPT-4o，直接走降级（用GPT-4o-mini代替）
→ 用户完全感知不到GPT-4o出问题了，因为有备用模型顶上

半开探测（HALF_OPEN，60秒后）：
请求N+1: 放行，尝试调用GPT-4o → 成功！
请求N+2~N+5: 都成功
→ 错误率恢复正常 → 回到 CLOSED，GPT-4o重新成为主力模型
```

## 六、降级 Fallback —— "首选餐厅关门了，去备选餐厅"

### 场景

你让 Claude Code 帮你"运行测试并修复失败的用例"。如果某个工具调用失败（比如某个 lint 工具没装），Claude Code 不会直接"罢工"说"我不干了"，而是会**尝试其他方式**继续完成任务——比如换一个检查工具，或者跳过这一步先完成主要任务，最后告诉你"这一步我跳过了，因为 XXX 工具没装"。

### 降级的核心思想

降级就是：**"最好的方案行不通时，按预案依次降低期望，但始终给用户一个结果，而不是什么都没有"**。

用一个生活场景理解：

```
你想吃宵夜，叫了个外卖：
Level 0（首选）: 楼下日式拉面店 —— 最想吃的
   ↓ 打电话过去，没人接（相当于"超时/熔断"）
Level 1（降级1）: 换成隔壁的兰州拉面 —— 同类型，稍微差一点但能接受
   ↓ 配送范围不到，无法下单
Level 2（降级2）: 自己泡一桶老坛酸菜面 —— 标准化、可控、随时可得
   ↓ 家里也没有泡面了
Level 3（降级3）: 喝点热水，配点饼干 —— 最低限度满足"吃点东西"的诉求
   ↓
Level 4（兜底）: "今晚先忍忍，明天早点起来吃早饭吧" —— 友好提示，管理预期
```

### 对应 Claude Code

```
任务：修复一个TypeScript类型错误

Level 0: 用最准确的方式——读取tsconfig.json，理解项目的类型系统，精准修复
   ↓ 如果项目配置复杂，模型读取上下文token太多导致变慢
Level 1: 降级——只读取报错文件的局部上下文，而不是整个项目
   ↓ 仍然无法确定正确的类型
Level 2: 降级——用 `as any` 或 `// @ts-ignore` 临时绕过，并明确告知用户"这是临时方案"
   ↓
Level 3: 兜底——告诉用户"这个类型错误比较复杂，建议你看一下XX文件，
         可能需要调整接口定义"
```

### 对应到你的业务（MoreTickets商品文案生成）

```
用户请求：用AI生成一段商品促销文案

Level 0: GPT-4o 生成（质量最高，成本最高）
   ↓ GPT-4o熔断中 / 超时
Level 1: GPT-4o-mini 生成（质量稍降，成本降低10倍）
   ↓ GPT-4o-mini也不可用
Level 2: 截断商品信息（只保留标题+价格），用本地小模型生成简单文案
   ↓ 本地模型服务也挂了
Level 3: 用预设的文案模板填空
        "【{商品名}】限时特惠仅售{价格}元，数量有限，先到先得！"
   ↓ 模板引擎也异常（极端情况）
Level 4: 友好提示
        "AI文案生成服务暂时繁忙，您可以稍后重试，
         或参考商品基础信息手动编辑"
```

**关键点**：每一层降级，用户得到的"价值"在递减，但**永远不会得到"系统崩溃/白屏/无响应"**这种最差体验。

## 七、综合案例：把六个概念串起来看一次完整请求

假设你在 Claude Code 里输入：

> "帮我检查一下 `src/api/order.ts` 这个文件有没有 bug，顺便跑一下测试"

完整流程（用本文的概念标注）：

```
1. 【限流】
   你的请求进入Claude Code → 检查是否超过当前会话的速率限制
   （正常情况下不会触发，除非你疯狂连续发消息）

2. 【超时】
   Claude决定第一步要读取 order.ts 文件
   → 文件读取工具调用，设置超时（比如10秒）
   → 文件很小，0.1秒就读完了 ✅

3. 【正常调用】
   Claude把文件内容发给模型分析
   → 调用Anthropic API（设置connect超时5s，read超时根据预估token估算）
   → 网络正常，2秒后返回分析结果 ✅

4. 【重试】（假设网络抖动）
   Claude决定运行测试命令 `npm test`
   → 调用Bash工具，但是Bash工具底层调用API记录日志时遇到网络抖动
   → 自动重试：等待1秒（+jitter）→ 第二次成功 ✅

5. 【超时 + 隔离】
   `npm test` 本身执行较久（比如要跑3分钟）
   → Bash工具有自己的超时配置（比如120秒）
   → 这个长时间运行的命令不会影响你同时让Claude
     "顺便看一下README"这种轻量操作（资源隔离）

6. 【熔断】（假设场景：Anthropic服务大面积故障）
   如果最近N次API调用持续失败（错误率超过阈值）
   → 熔断器打开，后续请求不再傻等超时
   → 直接提示"当前服务不可用，请稍后重试"

7. 【降级】（假设场景：主模型暂时不可用）
   → 自动切换到备用模型（如从Opus降级到Sonnet）
   → 继续完成任务，只是可能稍微"聪明程度"略降
   → 如果所有模型都不可用 → 友好提示，不会让你长时间空等
```

## 八、新手记忆口诀

| 概念 | 一句话记忆 |
|------|-----------|
| **超时 Timeout** | "等多久就该放手，别让一个请求拖垮全场" |
| **重试 Retry** | "网络抖一下可以再试，但要错开时间，错误参数别瞎重试" |
| **限流 Rate Limiting** | "排队进场，别让所有人同时冲门" |
| **隔离 Bulkhead** | "分舱设计，一个坑不会带倒所有人" |
| **熔断 Circuit Breaker** | "这条路最近老堵车，先别走，过会再探探路" |
| **降级 Fallback** | "首选没了走备选，备选没了走兜底，但永远要给个结果" |

## 九、给 Java 后端同学的延伸思考

Claude Code 作为"AI Agent"，它的每一次工具调用、每一次模型推理，本质上都是**对一个不稳定外部依赖的调用**——这和你写 Java 后端调用第三方支付接口、调用物流接口（比如 SF Express 顺丰 API）面临的问题是**一模一样的**：

```
顺丰API偶尔超时   ≈  Anthropic API偶尔超时
顺丰API限流       ≈  OpenAI API的TPM/RPM限制
顺丰API故障时     ≈  AI模型不可用时
   → 是不是要降级为"稍后人工处理"？  → 是不是要降级为低级模型/模板？
```

所以这套"限流-降级-熔断-超时-重试-隔离"的思维框架，**不仅适用于AI推理**，而是任何**调用不可控外部依赖**的系统都应该具备的"免疫系统"。

理解了 Claude Code 这种你天天用的工具背后的设计哲学，再去看 Resilience4j、Sentinel 的 API，你会发现——**它们只是把这些"常识"变成了配置参数和注解**而已。
