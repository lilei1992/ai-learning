Codex Agent Runtime

架构、记忆与扩展机制：从“会写代码的模型”到“可审计、可治理、可恢复的工程执行系统”。

Agent Loop Context Engineering AGENTS.md / Memories MCP / Skills / Plugins
开场定位：不要把 Codex 讲成聊天助手。它更像一个 coding agent runtime，用运行时边界把模型推理变成工程动作。
01 / Background

背景与问题

当模型开始修改代码、运行命令、调用 API，问题就不再只是“回答准不准”。

Context

它读到了什么？

项目规则、仓库文件、历史、工具说明、权限策略是否完整。

Action

它能做什么？

文件修改、Shell 命令、外部 API、生产操作必须受控。

Memory

它如何延续？

长任务需要恢复、复盘、摘要和经验沉淀。

Evidence

它怎么证明？

diff、测试结果、命令输出、审批记录要可追溯。

02 / Thesis

Codex 的核心不是 “模型会写代码”，而是 “模型在工程边界内循环执行任务”。

可靠的 agent 系统 = 模型推理 + 上下文构造 + 工具运行时 + 权限策略 + 状态记录 + 验证闭环。
03 / Core Concepts

核心概念

Agent

智能体

能感知上下文、调用工具、循环执行目标的系统。

Turn

轮次

一次用户输入到最终返回的完整执行周期。

Tool Call

工具调用

模型提出的结构化动作，执行权交给运行时。

Harness

运行框架

连接模型、工具、上下文、安全和状态的核心。

Rollout

执行轨迹

完整会话历史，可用于恢复、审计和记忆提炼。

Context

上下文窗口

有限资源，决定 agent 能安全理解多少信息。

04 / Agent Loop

实现原理：Agent Loop

Build Context合并规则、历史、工具、cwd、权限、memories。
Model Sample模型基于完整上下文做下一步推理。
Tool Calls输出 text 或结构化 tool call。
Execute路由、审批、沙箱、执行、捕获输出。
Record结果写回 thread，继续循环或结束。
关键点：模型不是直接执行动作，模型只提出意图；运行时决定能否执行、如何执行、如何记录。
05 / Prompt & Context

上下文构造：质量上限在 Context

Codex 每次推理都把必要状态放进请求中，服务端不依赖隐藏 session。

  • system：服务端控制的最高优先级指令
  • tools：模型可见工具定义
  • instructions：AGENTS.md、skills、配置
  • input：历史、工具结果、当前用户消息
system -> safety + product policy
tools -> exec / patch / browser / mcp
instructions -> AGENTS.md + skills
input -> history + current task
cache -> stable prefix matters
06 / Layered Architecture

架构分层：五层边界

SurfaceCLI / TUI / IDE / Web：输入、展示、审批交互。
ProtocolApp Server / JSON-RPC：thread、turn、event 的协议层。
RuntimeCore Session：上下文构建、模型调用、loop 控制。
ActionsTools / Sandbox / Approval：工具路由、权限、执行和输出捕获。
StateThread Store：transcript、rollout、compact、checkpoint。
07 / Tool Runtime

调用流程：从模型输出到真实动作

Model Outputtext / function_call / exec / patch。
ToolRouter解析响应项，转换为结构化工具调用。
Registry匹配工具 spec、handler 和参数约束。
Orchestrator审批、沙箱、重试、超时、错误处理。
Result输出进入 history，供下一次推理使用。
08 / Safety Runtime

安全不是提示词，是运行时策略

FS

文件权限

限制读写范围，避免误改无关路径。

NET

网络权限

控制外部访问，降低数据外泄风险。

APP

人工审批

高风险命令进入明确授权流程。

LOG

过程审计

命令、输出、错误、审批写入历史。

MCP 工具不是 Codex 原生 Shell 工具。第三方 MCP Server 必须自己实现权限、参数校验、敏感字段保护和审计。
09 / Memory System

记忆机制：静态规则 + 动态经验

Layer 1

AGENTS.md

人工维护。适合测试命令、架构边界、代码风格、禁改区域、权限要求。项目级文件可提交 Git，团队共享。

Layer 2

Memories

自动提炼。适合会话中涌现的 bug 模式、偏好、复盘知识。默认更像个人经验，不自动成为团队规范。

推荐工作流：让 Memories 自动积累，定期把稳定、可靠、团队可复用的内容提升到 AGENTS.md。
10 / Extensions

扩展机制：把能力工程化

MCP

连接外部系统

数据库、知识库、内部 API、发布平台变成受控工具。

Skills

固化工作流

代码审查、排障、文档生成、表格处理等 SOP。

Plugins

组合可安装能力

把技能、工具、应用封装为团队工具链。

Multi-Agent

并行子任务

适合独立模块调研，必须隔离上下文和写入范围。

CI/CD

交付链路

构建、测试、发布、回滚检查可作为受控工具。

Evaluation

质量评估

用回归测试、trace、成本延迟指标约束 agent 输出。

11 / Design Tradeoffs

关键设计取舍

取舍点 Codex 倾向 工程收益
无状态请求 每次携带完整上下文 可重放、可审计、支持 ZDR
Prompt Caching 保持静态前缀稳定 长对话成本接近线性增长
权限控制 运行时策略优先 不把安全寄托在模型“听话”上
记忆系统 AGENTS.md + Memories 分层 稳定规则和动态经验分开治理
12 / Code Example

示例代码：Agent Loop 伪代码

loop {
    let context = build_context(
        thread,
        project_rules,
        tools,
        memories,
        permissions,
    );

    let response = model.sample(context).await?;

    if response.is_assistant_message() {
        thread.record(response).await?;
        break;
    }

    let calls = tool_router.parse(response)?;
    let results = orchestrator.execute_with_policy(calls).await?;
    thread.record(results).await?;
}
13 / Team Setup

工程落地：先写好 AGENTS.md

# 项目协作规范

## 修改规则
- 先阅读相关代码和测试，再做最小安全修改。
- 不做无关重构，不重排无关文件。
- 不新增依赖，除非明确说明必要性和风险。

## 验证
- 修改后优先运行最相关测试。
- 如果测试未运行，必须说明原因。
- 不要声称测试通过，除非实际运行且退出码为 0。

## 安全
- 不打印 secrets、tokens、cookies、authorization headers。
- 支付、订单、库存、权限、生产写操作必须显式说明风险。
14 / Best Practices

最佳实践：小闭环胜过大承诺

任务拆小

一次只交付一个可验证目标。

验收前置

提前说明测试、页面、日志和 diff 检查。

工具最小权限

读写分离、范围限制、参数校验、脱敏。

保留执行证据

transcript、命令输出、测试结果、审批记录。

15 / FAQ & Mistakes

常见问题与误区

  • 误区：提示词写清楚就安全。
    修正：安全边界必须落在运行时策略。
  • 误区：Memories 可以替代 AGENTS.md。
    修正：团队稳定规范应提交到项目规则文件。
  • 误区：MCP 工具越多越好。
    修正：每个工具都增加权限、审计和缓存成本。
  • 误区：只看最终答案。
    修正：可信度来自执行证据和验证结果。
16 / Summary

总结：Codex 是工程运行时，不只是模型

Context

放大上下文

规则、历史、工具和验收标准越清晰，agent 越少猜。

Action

缩小动作半径

让每次修改可审查、可验证、可回滚。

Runtime

治理执行过程

权限、沙箱、审计、记忆和验证构成可信边界。

Q&A

问题讨论

可以从三个角度展开：怎么接入、怎么控风险、怎么沉淀团队知识。
