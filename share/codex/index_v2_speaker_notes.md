## 0. 分享前准备

### 0.1 一句话主线

这场分享要反复强调一句话：

> Codex 的核心不是模型会写代码，而是模型在工程边界内循环执行任务。

如果听众只记住一件事，应是：

- 模型负责推理和提出动作。
- 运行时负责决定动作能否执行、如何执行、如何记录。
- 团队要建设的是上下文、权限、工具、记忆和验证体系。

### 0.2 建议讲述顺序

1. 先破除误解：Codex 不是代码补全，也不是普通 Chatbot（聊天机器人 / 对话式模型）。
2. 再建立模型：Agent Loop（智能体循环）是核心执行闭环。
3. 然后拆分系统：Context（上下文）、Tool Runtime（工具运行时）、Safety Runtime（安全运行时）、Memory（记忆系统）、Extensions（扩展机制）。
4. 最后落到团队实践：AGENTS.md（项目级智能体规则文件）、MCP（Model Context Protocol，模型上下文协议）权限、任务拆分、执行证据、评估体系。

### 0.3 面向不同听众的侧重点

| 听众 | 应重点讲 | 可略讲 |
|---|---|---|
| 后端开发 | Agent Loop（智能体循环）、Tool Runtime（工具运行时）、安全权限、MCP（模型上下文协议）、CI/CD（持续集成 / 持续交付） | 视觉设计、演示控件 |
| 前端开发 | IDE（集成开发环境）接入、App Server（应用服务）、Context（上下文）、代码审查流程 | ZDR（Zero Data Retention，零数据留存）、Prompt Caching（提示词缓存）细节 |

## 1. 封面页：Codex Agent Runtime（Codex 智能体运行时）

### 本页目的

建立整场分享的定位：这不是工具使用技巧，而是一次关于 AI coding agent runtime（AI 编程智能体运行时）的架构分享。

### 幻灯片内容

- 标题：`Codex Agent Runtime`（Codex 智能体运行时）
- 副标题：架构、记忆与扩展机制
- 关键词：`Agent Loop`（智能体循环）、`Context Engineering`（上下文工程）、`AGENTS.md / Memories`（项目规则 / 记忆条目）、`MCP / Skills / Plugins`（模型上下文协议 / 技能 / 插件）

### 讲解重点

开场不要直接说“Codex 很强，可以帮我们写代码”。这会把听众带到工具演示的心智里。更好的开场是：

> 今天我们不把 Codex 当成一个聊天窗口，而是把它拆成一套工程运行时。它背后包含模型推理、上下文构造、工具调用、安全策略、会话状态、记忆系统和扩展协议。

Codex 的输出不是纯文本，而可能直接变成：

- 文件修改。
- Shell（命令行外壳）命令。
- 测试执行。
- PR（Pull Request，合并请求）说明。
- 浏览器操作。
- MCP（模型上下文协议）工具调用。

这意味着它进入了真实工程系统，因此必须讨论边界、审计、恢复和治理。

### 可用类比

可以把普通 LLM（Large Language Model，大语言模型）应用类比为“顾问”，它给建议；把 Codex 类比为“受权限管理的实习工程师”，它可以进入仓库执行任务，但必须遵守团队规则、权限边界和验收流程。

### 转场话术

“既然 Codex 能做的不只是回答问题，那我们首先要问：当一个模型开始改代码、跑命令、调用工具时，工程上真正的问题是什么？”

## 2. 背景与问题

### 本页目的

说明 Agent（智能体）化之后，问题从“答案质量”升级为“工程行为治理”。

### 幻灯片四个问题

1. 它读到了什么？
2. 它能做什么？
3. 它如何延续？
4. 它怎么证明？

### 讲解重点

普通 Chatbot（聊天机器人 / 对话式模型）的风险主要是“说错”。Coding agent（编程智能体）的风险更复杂，因为它能行动：

- 如果上下文不完整，它可能基于猜测修改代码。
- 如果权限过大，它可能执行不该执行的命令。
- 如果历史不可恢复，长任务中断后很难继续。
- 如果没有证据，团队无法判断它是否真的完成了任务。

这四个问题对应四类工程能力：

| 问题 | 对应能力 |
|---|---|
| 它读到了什么 | Context Engineering（上下文工程） |
| 它能做什么 | Tool Runtime / Permission（工具运行时 / 权限） |
| 它如何延续 | Thread / Rollout / Memory（会话线程 / 执行轨迹 / 记忆） |
| 它怎么证明 | Test / Diff / Audit（测试 / 差异 / 审计） |

### 延展知识

“上下文”不是只指 prompt（提示词 / 模型输入）。对 coding agent（编程智能体）来说，上下文包括：

- 仓库结构。
- 当前工作目录。
- 用户需求。
- 历史对话。
- 已运行命令的输出。
- 代码搜索结果。
- 项目规则文件。
- 可用工具定义。
- 权限和沙箱策略。
- 长对话摘要。

### 可能追问

**问：为什么不让模型自己判断风险？**  
答：模型判断可以作为辅助，但不能成为安全边界。安全边界必须由确定性的运行时策略控制，例如文件系统权限、网络权限、审批模式和审计日志。

### 转场话术

“所以我们看 Codex，不能只看模型能力，而要看它如何把模型能力约束在工程边界内。”

## 3. 核心论点

### 本页目的

用一句话把整场分享钉住：Codex 是“模型 + 工程边界 + 执行闭环”。

### 核心句

> Codex 的核心不是“模型会写代码”，而是“模型在工程边界内循环执行任务”。

### 讲解重点

这句话有三个关键词：

1. **模型**：负责理解需求、推理下一步、生成工具调用意图。
2. **工程边界**：包括仓库规则、权限、沙箱、审批、测试和审计。
3. **循环执行**：不是一次生成答案，而是多次“推理 -> 行动 -> 观察 -> 再推理”。

可靠的 agent（智能体）系统通常由以下部分构成：

- 模型推理。
- 上下文构造。
- 工具运行时。
- 权限策略。
- 状态记录。
- 验证闭环。

### 可用例子

让 agent（智能体）修复一个测试失败问题：

1. 读取错误日志。
2. 搜索相关代码。
3. 判断可能原因。
4. 修改最小代码。
5. 运行相关测试。
6. 如果失败，继续读取输出并迭代。
7. 通过后总结变更和风险。

这不是一次 prompt（提示词 / 模型输入）就结束，而是一个 loop（循环）。

“这个循环到底由哪些基本概念组成？下一页先统一术语。”

## 4. 核心概念

### 本页目的

统一后续讲解需要用到的术语，避免听众把 thread（会话线程）、turn（轮次 / 回合）、tool call（工具调用）混在一起。

### 逐项讲解

#### Agent（智能体）

Agent（智能体）不是“更会聊天的模型”。它至少具备三个特征：

- 能接收上下文。
- 能选择下一步动作。
- 能调用工具并基于结果继续行动。

#### Turn（轮次 / 回合）

Turn（轮次 / 回合）是一次用户输入触发的完整执行周期。一个 turn 里可能包含多次模型调用和多次工具调用。

例如用户说“修复这个测试”，一个 turn（轮次）内可能发生：

- 模型分析。
- `rg` 搜索。
- `sed` 查看文件。
- `apply_patch` 修改。
- `npm test` 验证。
- 模型总结。

#### Tool Call（工具调用）

Tool Call（工具调用）是模型提出的结构化动作请求。注意不是模型真的执行，而是运行时解析后执行。

#### Harness（运行框架 / 控制器）

Harness（运行框架 / 控制器）可以理解为把模型、工具、上下文、权限、状态串起来的控制器。它不是单个 UI（用户界面），也不是单个 prompt（提示词 / 模型输入），而是 agent（智能体）的生命周期管理层。

#### Rollout（执行轨迹）

Rollout（执行轨迹）是完整执行过程记录。它对于恢复、审计、复盘、记忆提炼都很关键。

#### Context（上下文）

Context（上下文）是模型本次推理能看到的全部信息。它是有限资源，必须管理。

### 常见混淆

| 概念 | 容易误解 | 正确理解 |
|---|---|---|
| Turn（轮次） | 一次模型调用 | 一次用户请求的完整执行周期 |
| Tool Call（工具调用） | 模型直接执行函数 | 模型提出动作，运行时执行 |
| Context（上下文） | 用户写的 prompt | 所有进入模型输入的信息 |
| Memory（记忆） | 聊天历史 | 静态规则 + 动态经验 + 摘要状态 |

### 转场话术

“有了术语之后，我们看最核心的执行机制：Agent Loop（智能体循环）。”

## 5. 实现原理：Agent Loop（智能体循环）

### 本页目的

解释 Codex 如何从“回答一次”变成“持续工作”。

### 五步流程

1. **Build Context（构建上下文）**：组装本轮推理需要的信息。
2. **Model Sample（模型采样）**：模型基于上下文推理下一步。
3. **Tool Calls（工具调用）**：模型输出文本或结构化工具调用。
4. **Execute（执行）**：运行时执行工具。
5. **Record（记录）**：写回历史，继续循环或结束。

### 讲解重点

Agent Loop（智能体循环）的关键不是“模型很聪明”，而是模型每一步都能观察新信息。

一次典型循环：

```text
用户目标
  -> 模型判断需要看代码
  -> 调用搜索工具
  -> 搜索结果写回上下文
  -> 模型判断需要改文件
  -> 调用 patch 工具
  -> patch 结果写回上下文
  -> 模型判断需要验证
  -> 调用测试命令
  -> 测试输出写回上下文
  -> 模型总结最终结果
```

### 关键设计点

工具调用结果必须写回上下文，否则模型无法基于真实观察继续推理。  
这正是 ReAct（Reason + Act + Observe，推理 + 行动 + 观察）模式的工程化形态。

### 深入理解

Agent Loop（智能体循环）的稳定性取决于三个条件：

- 输入是否足够真实：是否真的读取了代码、日志、测试结果。
- 动作是否受控：是否经过权限、沙箱、审批。
- 终止是否可靠：是否知道什么时候完成，什么时候继续，什么时候停下来问人。

### 转场话术

“Agent Loop（智能体循环）每一步都依赖上下文。那么上下文到底是如何构造的？”

## 6. Prompt（提示词）与 Context（上下文）

### 本页目的

说明 Codex 的推理质量取决于 Context Engineering（上下文工程），而不是单纯“prompt（提示词）写得漂亮”。

### 四层输入

#### system（系统指令层）

最高优先级，由服务端或产品控制。通常包含安全、产品行为和底层约束。

#### tools（工具定义层）

工具定义。告诉模型“你可以请求哪些动作”，包括工具名、参数结构、能力边界。

#### instructions（开发者 / 项目指令层）

开发者或项目注入的指令，例如：

- `AGENTS.md`（项目级智能体规则文件）
- skills（技能）
- 插件说明
- 全局偏好
- 当前环境说明

#### input（当前输入层）

对话历史、工具调用结果、当前用户消息。

### 讲解重点

Codex 的一个重要设计是：每次推理都携带完整必要上下文，而不是依赖服务端隐藏 session（会话状态）。这样做带来几个收益：

- 更容易重放和调试。
- 更符合企业零数据留存需求。
- 可以支持兼容 Responses API（响应接口）的不同后端。

代价是 prompt（提示词 / 模型输入）会持续增长，因此需要 Prompt Caching（提示词缓存）和 Context Compaction（上下文压缩）。

### Prompt Caching（提示词缓存）解释

可以用“编译缓存”类比。  
如果 prompt（提示词 / 模型输入）前半部分保持不变，服务端可以复用之前的 KV-Cache（键值缓存），减少重复计算。

因此 Codex 倾向于：

- 把稳定内容放前面。
- 把动态内容追加到后面。
- 避免中途修改早期消息。
- 避免工具列表顺序乱变。

### 讲给后端的类比

Prompt（提示词 / 模型输入）可以类比为一次完整 HTTP request body（HTTP 请求体）。  
无状态设计类似 stateless JWT（无状态 JSON Web Token）：每次请求都自包含，不依赖服务器 session（会话状态）。

### 转场话术

“上下文只是输入。要让系统可维护，还需要把 UI（用户界面）、协议、运行时、工具和状态分层。”

## 7. 架构分层：五层边界

### 本页目的

让听众建立 Codex 的系统架构地图。

### 五层说明

#### Surface（交互入口层）

面向用户的入口，例如 CLI（命令行界面）、TUI（终端用户界面）、IDE（集成开发环境）、Web（网页端）。  
它负责输入、展示、审批交互，但不应该把核心 agent（智能体）逻辑写死在 UI（用户界面）层。

#### Protocol（协议层）

协议层，例如 App Server（应用服务）/ JSON-RPC（基于 JSON 的远程过程调用协议）。  
它把 thread（会话线程）、turn（轮次）、event（事件）抽象出来，让多个客户端共享一套 runtime（运行时）。

#### Runtime（运行时层）

核心运行时。负责：

- 构建上下文。
- 调用模型。
- 控制 Agent Loop（智能体循环）。
- 决定何时继续、停止或请求用户输入。

#### Actions（动作层）

动作层。负责工具路由、审批、沙箱、命令执行和输出捕获。

#### State（状态层）

状态层。负责 transcript（对话记录）、rollout（执行轨迹）、compact（压缩）、checkpoint（检查点）、恢复和复盘。

### 为什么要分层

如果没有分层，常见问题是：

- UI（用户界面）和模型调用耦合，换一个入口就要重写。
- 工具调用散落各处，权限无法统一治理。
- 会话历史不可复盘，失败后很难定位原因。
- 扩展工具越来越多，系统变成不可控脚本集合。

### 架构价值

分层之后：

- UI（用户界面）可以替换。
- 工具可以扩展。
- 权限可以统一。
- 状态可以恢复。
- 审计可以贯穿全链路。

### 转场话术

“架构里最容易被低估的是动作层。我们看模型输出如何变成真实动作。”

## 8. Tool Runtime（工具运行时）：从模型输出到真实动作

### 本页目的

解释 Tool Calling（工具调用）不是模型直接调用函数，而是运行时管理的动作流水线。

### 五段流程

#### Model Output（模型输出）

模型输出可能是普通文本，也可能是结构化调用，例如：

- `function_call`（函数调用）
- `exec`（命令执行）
- `patch`（补丁修改）
- browser action（浏览器动作）
- MCP tool call（模型上下文协议工具调用）

#### ToolRouter（工具路由器）

解析模型响应，把响应项转换为内部工具调用对象。

#### Registry（工具注册表）

查找工具定义，匹配 handler（处理器），校验参数结构。

#### Orchestrator（编排器）

真正的安全中枢，处理：

- 审批。
- 沙箱。
- 超时。
- 重试。
- 错误归类。
- 输出截断。

#### Result（执行结果）

执行结果写回 history（历史记录），让模型基于真实结果继续推理。

### 重要边界

模型不能绕过运行时直接执行动作。  
这意味着 agent（智能体）系统的安全性不只取决于模型，也取决于工具层实现。

### 可讲示例

用户让 Codex “删除无用文件”：

1. 模型可能提出 `rm` 命令。
2. ToolRouter（工具路由器）解析出命令调用。
3. Orchestrator（编排器）判断这是 destructive action（破坏性动作）。
4. 根据 approval mode（审批模式）要求用户确认。
5. sandbox（沙箱）限制只能操作工作区。
6. 输出写回 history（历史记录）。

### 转场话术

“一旦工具能触碰真实文件和命令，安全就不能靠提示词解决。”

## 9. Safety Runtime（安全运行时）：安全不是提示词

### 本页目的

强调安全边界必须在运行时，而不是只写在 prompt（提示词 / 模型输入）中。

### 四类安全机制

#### 文件权限

限制读写路径，避免 agent（智能体）误改仓库外、系统目录或敏感文件。

#### 网络权限

控制外部访问，降低数据外泄和 SSRF（Server-Side Request Forgery，服务端请求伪造）风险。

#### 人工审批

高风险操作需要显式授权。典型包括：

- 删除文件。
- 批量修改。
- 安装依赖。
- 访问网络。
- 生产写操作。
- 支付、订单、权限变更。

#### 过程审计

记录命令、输出、错误、审批和最终变更。

### MCP（模型上下文协议）安全重点

必须明确说明：

> Codex 原生 Shell（命令行外壳）工具可以被 Codex 的 sandbox / approval（沙箱 / 审批）管控，但第三方 MCP（Model Context Protocol，模型上下文协议）工具需要 MCP Server（MCP 服务端）自己实现权限边界。

一个不安全的 MCP（模型上下文协议）工具可能拥有过大权限，例如：

- 直接查询生产数据库。
- 直接修改订单状态。
- 直接触发发布。
- 返回未脱敏的用户数据。

### 工程建议

设计 MCP（模型上下文协议）工具时要做到：

- 读写工具分离。
- 参数白名单。
- 继承调用人身份或使用最小权限服务账号。
- 返回值脱敏。
- 审计调用人、参数摘要、结果摘要、耗时和失败原因。
- 高风险动作走二次确认。

### 转场话术

“安全解决的是动作边界。长任务还需要解决另一个问题：如何记住、恢复和复盘。”

## 10. Memory System（记忆系统）：静态规则 + 动态经验

### 本页目的

解释 Codex 记忆机制的双层模型，以及它如何服务团队协作。

### Layer 1（第一层）：AGENTS.md（项目级智能体规则文件）

`AGENTS.md`（项目级智能体规则文件）是人工维护的静态规则层。适合写：

- 测试命令。
- 构建命令。
- 代码风格。
- 架构边界。
- 禁改区域。
- 安全要求。
- 依赖管理规则。
- PR（Pull Request，合并请求）/ 提交要求。

它的优势：

- 可读。
- 可审查。
- 可提交 Git。
- 团队共享。
- 规则稳定。

### Layer 2（第二层）：Memories（记忆条目）

Memories（记忆条目）是自动提炼的动态经验层。适合沉淀：

- 会话中发现的 bug 模式。
- 用户偏好。
- 临时技术决策。
- 复盘结论。
- 某些仓库的隐性经验。

它的限制：

- 默认更偏个人本地经验。
- 不一定适合直接作为团队规范。
- 需要定期审查和提升。

### 推荐工作流

1. 用 `AGENTS.md`（项目级智能体规则文件）建立稳定基线。
2. 让 Memories（记忆条目）自动积累实际经验。
3. 定期 review（审查）Memories（记忆条目）。
4. 把可靠、稳定、团队通用的内容迁移到 `AGENTS.md`（项目级智能体规则文件）。

### 关键提醒

Memories（记忆条目）不是万能的。长期稳定的团队约定不应该只存在于某个人的记忆系统里，而应该进入项目规则文件。

### 转场话术

“当规则和记忆稳定之后，我们会希望把更多内部系统接进来，这就进入扩展机制。”

## 11. Extensions（扩展机制）：把能力工程化

### 本页目的

说明 Codex 的扩展不是“接更多工具”这么简单，而是把能力纳入治理体系。

### MCP（Model Context Protocol，模型上下文协议）

MCP（模型上下文协议）用来把外部系统变成模型可见工具，例如：

- 数据库查询。
- 内部知识库。
- 监控系统。
- 发布系统。
- 工单系统。
- 代码仓库。

重点不是“能连”，而是“受控地连”。

### Skills（技能）

Skills（技能）用来固化工作流。例如：

- 系统化排障。
- 代码审查。
- TDD（Test-Driven Development，测试驱动开发）。
- 文档生成。
- 表格分析。
- 演示文稿制作。

它适合把团队经验变成可重复执行的 SOP（Standard Operating Procedure，标准作业流程）。

### Plugins（插件）

Plugins（插件）可以组合 skills（技能）、tools（工具）、apps（应用），把一组能力打包成可安装能力。

### Multi-Agent（多智能体）

适合：

- 多模块并行阅读。
- 独立任务调研。
- 大量资料归纳。

但要避免多个 agent（智能体）同时写同一文件或共享不一致状态。

### CI/CD（持续集成 / 持续交付）

可以接入：

- 构建。
- 测试。
- 发布检查。
- 回滚检查。

但生产写操作必须保守处理。

### Evaluation（评估）

Agent（智能体）需要评估体系，包括：

- 回归测试。
- trace（执行追踪）观测。
- 成本。
- 延迟。
- 人工体验。
- 失败模式归因。

### 转场话术

“这些机制背后有一些重要设计取舍。理解取舍，比记住功能点更重要。”

## 12. 关键设计取舍

### 本页目的

帮助听众理解 Codex 为什么这么设计，而不是只知道它有什么功能。

### 取舍 1：无状态请求

Codex 倾向于每次携带完整上下文，而不是依赖服务端隐藏 session（会话状态）。

收益：

- 可重放。
- 可审计。
- 易调试。
- 支持 ZDR（Zero Data Retention，零数据留存）。

代价：

- prompt（提示词 / 模型输入）会变长。
- 需要缓存和压缩。

### 取舍 2：Prompt Caching（提示词缓存）

为了降低长对话成本，需要保持 prompt（提示词 / 模型输入）静态前缀稳定。

这解释了为什么：

- 工具列表要稳定。
- 配置变更尽量追加，而不是修改历史。
- 动态时间戳不适合放在静态 instruction（指令）里。

### 取舍 3：权限控制

Codex 不把安全寄托在“模型会遵守提示词”上，而是依赖运行时策略。

收益：

- 行为可控。
- 高风险动作可审批。
- 失败可审计。

### 取舍 4：记忆系统分层

`AGENTS.md`（项目级智能体规则文件）和 Memories（记忆条目）分层，可以把稳定规则和动态经验分开治理。

### 可讲总结

这些取舍都围绕一个目标：

> 在不牺牲可审计性和安全性的前提下，让 agent（智能体）可以完成更长、更复杂的工程任务。

“下面用一段伪代码，把前面所有概念串起来。”

## 13. 示例代码：Agent Loop（智能体循环）伪代码

### 本页目的

用代码形式把 Agent Loop（智能体循环）的控制流具体化。

### 代码讲解

```rust
loop {
    let context = build_context(
        thread,
        project_rules,
        tools,
        memories,
        permissions,
    );

    let response = model.sample(context).await?;

    if response.is_assistant_message() {
        thread.record(response).await?;
        break;
    }

    let calls = tool_router.parse(response)?;
    let results = orchestrator.execute_with_policy(calls).await?;
    thread.record(results).await?;
}
```

### 逐行解释

#### `build_context`

把所有必要上下文组装起来：

- 当前 thread（会话线程）。
- 项目规则。
- 工具定义。
- memories（记忆条目）。
- 权限策略。

#### `model.sample`

调用模型，让模型基于当前上下文判断下一步。

#### `is_assistant_message`

如果模型输出最终回答，说明本轮 turn（轮次）可以结束。

#### `tool_router.parse`

如果模型输出工具调用，解析为内部结构化调用。

#### `orchestrator.execute_with_policy`

执行工具，但要经过权限、审批、沙箱和错误处理。

#### `thread.record`

把结果写回历史，供下一次推理使用。

### 讲解技巧

这一页不要陷入 Rust（系统编程语言）语法。重点是控制流：

> 每一次工具结果都会变成下一次模型推理的输入。

### 转场话术

“理解运行机制后，团队落地第一步不是写复杂插件，而是写好项目规则。”

## 14. 工程落地：先写好 AGENTS.md（项目级智能体规则文件）

### 本页目的

把抽象架构落到团队可以马上执行的动作上。

### 为什么 AGENTS.md（项目级智能体规则文件）重要

AGENTS.md（项目级智能体规则文件）是 agent（智能体）进入仓库前要读的团队手册。它越清晰，agent 越少猜。

适合写进去的内容：

- 修改前必须阅读哪些目录。
- 常用测试命令。
- 构建命令。
- 不允许修改的区域。
- 添加依赖规则。
- API（Application Programming Interface，应用程序接口）兼容要求。
- 数据库和消息格式风险。
- 安全要求。

### 示例解读

#### 修改规则

强调“小改动”和“先理解后修改”。这能减少 agent（智能体）为了完成目标而做大范围重构。

#### 验证

强调“不能声称测试通过，除非实际运行”。这对 agent（智能体）协作非常关键，因为团队信任来自证据。

#### 安全

明确禁止打印敏感信息，并对支付、订单、库存、权限等高风险领域提醒。

### AGENTS.md（项目级智能体规则文件）写作建议

写得越具体越好：

```markdown
不好：修改后跑测试。
更好：修改 payment 模块后运行 `mvn -pl payment -Dtest=RefundRetryServiceTest test`。
```

```markdown
不好：注意安全。
更好：禁止在日志中输出 access_token、Authorization、Cookie、用户手机号和身份证号。
```

### 转场话术

“有了规则之后，日常协作还要靠实践原则来降低风险。”

## 15. 最佳实践：小闭环胜过大承诺

### 本页目的

给团队一套可落地的操作原则。

### 实践 1：任务拆小

不要让 agent（智能体）一次做“重构整个支付系统”。更好的任务是：

- 修复一个测试。
- 为一个函数补充边界处理。
- 为一个 API（应用程序接口）增加一个校验。
- 给一个模块补一组单测。

### 实践 2：验收前置

在任务开始前告诉 agent（智能体）：

- 修改哪些文件。
- 跑哪些测试。
- 看哪些页面。
- 检查哪些日志。
- 哪些行为不能变。

### 实践 3：工具最小权限

工具越强，风险越大。要做：

- 读写分离。
- 参数校验。
- 范围限制。
- 返回值脱敏。
- 高风险二次确认。

### 实践 4：保留执行证据

最终交付至少应包含：

- 改了哪些文件。
- 为什么这么改。
- 跑了什么命令。
- 测试结果是什么。
- 还有什么风险。

### 团队落地建议

可以建立一套 agent task（智能体任务）模板：

```markdown
目标：
范围：
禁止修改：
验收命令：
兼容要求：
风险点：
```

### 转场话术

“如果没有这些实践，很容易踩进几个常见误区。”

## 16. 常见问题与误区

### 本页目的

提前纠正常见错误理解。

### 误区 1：提示词写清楚就安全

提示词只能影响模型行为倾向，不能成为安全控制。  
真正的安全边界必须是确定性的：

- 权限。
- 沙箱。
- 审批。
- 审计。

### 误区 2：Memories（记忆条目）可以替代 AGENTS.md（项目级智能体规则文件）

Memories（记忆条目）是动态经验，不适合替代团队规范。  
团队规范必须可读、可审查、可版本管理。

### 误区 3：MCP（模型上下文协议）工具越多越好

工具越多，模型选择复杂度越高，权限面越大，审计成本越高，缓存稳定性也可能受影响。

### 误区 4：只看最终答案

Agent（智能体）的可信度来自过程证据：

- 它读了哪些文件。
- 它运行了哪些命令。
- 它改了什么。
- 它如何验证。
- 它是否处理失败。

### 可补充误区

#### 误区：让 agent（智能体）一次完成完整需求更高效

实际通常相反。大任务导致上下文漂移、错误累积、验证困难。

#### 误区：full-auto（全自动模式）最省事

full-auto（全自动模式）只适合低风险、沙箱或实验环境。业务关键系统不应默认使用。

### 转场话术

“最后我们把整场分享收束成三个动作：放大上下文，缩小动作半径，治理执行过程。”

## 17. 总结

### 本页目的

让听众带走清晰、可执行的结论。

### 三个结论

#### 放大上下文

让 agent（智能体）少猜：

- 写清 AGENTS.md（项目级智能体规则文件）。
- 指定验收标准。
- 提供错误日志。
- 提供边界条件。
- 提供兼容要求。

#### 缩小动作半径

让每次动作可审查：

- 小任务。
- 小 diff（差异）。
- 小范围测试。
- 明确禁止修改范围。

#### 治理执行过程

让结果可信：

- 权限。
- 沙箱。
- 审批。
- 审计。
- 记忆。
- 验证。

### 最后一段话

Codex 的价值不是替代工程师，而是把模型能力接入工程体系。  
团队真正要建设的不是“神奇提示词”，而是上下文、工具、权限、记忆和验证体系。

# 问题讨论备答

## A. 关于定位与价值

### Q1：Codex 和普通 ChatGPT 写代码有什么本质区别？

普通 ChatGPT 主要输出建议或代码片段，执行动作需要人复制粘贴。Codex 的关键区别是它能进入真实工程环境，读取文件、修改文件、运行命令、调用工具，并把执行结果继续放回上下文中推理。  
所以 Codex 更像一个 coding agent runtime（编程智能体运行时），而不是单纯的代码生成器。

### Q2：Codex 会不会替代工程师？

更现实的理解是：Codex 替代一部分重复性工程动作，但不会替代工程判断。  
它适合做检索、局部修改、测试补充、文档整理、重复排查等任务。架构取舍、业务语义、风险判断、上线责任仍然需要工程师把关。

### Q3：为什么把 Codex 称为 runtime（运行时）？

因为它不只是模型调用。它包含：

- 生命周期控制。
- 上下文组装。
- 工具路由。
- 权限审批。
- 沙箱执行。
- 历史记录。
- 记忆加载。
- 结果验证。

这些都是运行时能力。

## B. 关于 Agent Loop（智能体循环）

### Q4：Agent Loop（智能体循环）和普通多轮对话有什么区别？

普通多轮对话主要是文本输入输出。Agent Loop（智能体循环）中间会插入工具调用，并把工具结果写回上下文。  
也就是说，模型不是只“想”，还会通过运行时“观察真实世界”，例如代码文件、测试输出、终端日志。

### Q5：Agent Loop（智能体循环）什么时候停止？

通常在模型输出最终 assistant message（助手消息）时停止。工程上还需要额外 stop condition（停止条件），例如：

- 达到工具调用次数上限。
- 达到 token（令牌 / 模型计费与长度单位）或成本上限。
- 多次失败需要用户介入。
- 触发高风险操作等待审批。
- 当前验收标准已满足。

### Q6：如果模型一直循环怎么办？

需要运行时控制：

- 最大步数。
- 超时。
- 失败计数。
- 重试策略。
- 让模型总结当前状态并请求用户确认。

不能完全依赖模型自己停下来。

## C. 关于 Context（上下文）与 Prompt（提示词）

### Q7：为什么说质量上限在 Context（上下文），而不在 Prompt（提示词）？

Prompt（提示词）写得再好，如果模型没有读到项目规则、代码结构、错误日志、测试命令和业务边界，它只能猜。  
高质量上下文能让模型基于事实行动，而不是基于想象行动。

### Q8：上下文越多越好吗？

不是。上下文太多会带来：

- token（令牌 / 模型计费与长度单位）成本上升。
- 重要信息被稀释。
- 噪声干扰判断。
- 触发 context window（上下文窗口）限制。

好的上下文工程不是“全塞进去”，而是“放入当前任务需要的最小充分信息”。

### Q9：Prompt Caching（提示词缓存）为什么重要？

长对话里，每次请求都重新计算全部历史成本很高。Prompt Caching（提示词缓存）可以复用稳定前缀的计算结果，让新增内容的成本更低。  
因此要保持系统指令、工具定义、项目规则等静态前缀稳定。

### Q10：哪些行为容易破坏缓存？

常见包括：

- 中途切换模型。
- 改变工具列表。
- 改变 sandbox（沙箱）或 approval（审批）配置。
- 改变工作目录。
- 在静态指令中加入动态时间戳。
- 重写早期历史消息。

## D. 关于安全与权限

### Q11：提示词里写“不要执行危险命令”不够吗？

不够。提示词属于软约束，安全必须用硬边界实现。  
例如文件系统权限、网络权限、审批模式、命令 denylist（拒绝列表）、沙箱和审计日志。

### Q12：什么时候可以使用 full-auto（全自动模式）？

建议只在以下场景使用：

- 沙箱环境。
- 低风险练习仓库。
- 无生产数据。
- 任务边界非常明确。
- 有自动测试和可回滚机制。

业务核心系统、支付、订单、权限、生产运维不建议默认使用 full-auto（全自动模式）。

### Q13：MCP（模型上下文协议）工具为什么风险更高？

因为 MCP Server（MCP 服务端）可能直接连接内部系统。如果工具设计不当，agent（智能体）可能间接拥有过大权限。  
Codex 的 shell sandbox（命令行沙箱）不会自动保护第三方 MCP（模型上下文协议）工具，所以 MCP Server（MCP 服务端）必须自己实现 guardrails（防护栏 / 安全约束）。

### Q14：如何设计安全的 MCP（模型上下文协议）工具？

建议：

- 工具粒度小。
- 读写分离。
- 参数白名单。
- 返回值脱敏。
- 最小权限账号。
- 高风险操作二次确认。
- 记录调用审计。
- 明确 timeout（超时）和 retry（重试）。
- 默认拒绝未知参数。

### Q15：如何处理敏感信息？

原则：

- 不把 secrets（密钥 / 敏感凭据）写入 AGENTS.md（项目级智能体规则文件）。
- 不让工具返回完整 token（令牌）、cookie、authorization header（授权请求头）。
- 日志中做脱敏。
- 生产数据只返回必要字段。
- 对用户数据使用最小可见范围。

## E. 关于记忆机制

### Q16：AGENTS.md（项目级智能体规则文件）和 Memories（记忆条目）的区别是什么？

`AGENTS.md`（项目级智能体规则文件）是人工维护、稳定、可提交 Git 的项目规则。  
Memories（记忆条目）是自动提炼的动态经验，通常更偏个人本地记忆。

一句话：

- AGENTS.md（项目级智能体规则文件）管“团队明确规则”。
- Memories（记忆条目）管“历史会话经验”。

### Q17：Memories（记忆条目）可以团队共享吗？

通常不应该直接依赖 Memories（记忆条目）作为团队共享机制。  
更好的做法是定期 review（审查）Memories，把稳定可靠的知识提升到 `AGENTS.md`（项目级智能体规则文件）或团队 wiki（知识库）。

### Q18：AGENTS.md（项目级智能体规则文件）应该写多详细？

写到 agent（智能体）能减少猜测为止。推荐具体、可执行、可验证：

- 明确命令。
- 明确目录。
- 明确禁止项。
- 明确风险流程。

不要写空泛原则，例如“注意代码质量”。应该写具体规则。

### Q19：AGENTS.override.md（项目规则覆盖文件）什么时候用？

适合强制覆盖上层规则，例如某个子模块使用完全不同测试命令，或临时高风险目录需要更严格权限。  
不建议滥用，因为 override（覆盖规则）会降低规则可预期性。

## F. 关于工程落地

### Q20：团队第一步应该做什么？

建议从三件事开始：

1. 给核心仓库写 `AGENTS.md`（项目级智能体规则文件）。
2. 定义 agent（智能体）任务模板。
3. 明确高风险动作审批规则。

不要第一步就做复杂 MCP（模型上下文协议）平台。

### Q21：哪些任务最适合先交给 Codex？

适合：

- 小 bug（缺陷）修复。
- 单测补充。
- 文档整理。
- 日志分析。
- 小范围重构。
- 迁移脚本草稿。
- 重复性代码修改。

不适合一开始就交给：

- 跨多个核心模块的大重构。
- 业务语义不清的需求。
- 生产数据写操作。
- 缺少测试的高风险改动。

### Q22：如何写一个好的 Codex 任务？

推荐模板：

```markdown
目标：
范围：
禁止修改：
上下文：
验收命令：
兼容要求：
风险点：
输出要求：
```

### Q23：如何判断 agent（智能体）的结果可信？

看证据，而不是看语气：

- 是否说明改了哪些文件。
- 是否解释为什么这么改。
- 是否运行测试。
- 是否贴出命令和结果。
- 是否说明未验证部分。
- 是否保留兼容性。
- 是否没有扩大范围。

### Q24：如果 agent（智能体）没跑测试但说完成了怎么办？

不能接受“完成”结论。应要求：

- 明确测试未运行原因。
- 补跑相关测试。
- 或说明人工验证步骤。

团队规则里应明确：没有验证证据不能声称通过。

## G. 关于扩展与平台化

### Q25：什么时候需要 MCP（模型上下文协议）？

当 agent（智能体）需要稳定访问外部系统，且这种访问会重复发生时，适合做 MCP（模型上下文协议）。  
例如查询内部文档、读取工单、查监控、触发测试环境构建。

如果只是一次性操作，不一定要做 MCP（模型上下文协议），可以先用人工输入或脚本。

### Q26：什么时候需要 Skills（技能）？

当某类任务有稳定流程时，例如：

- code review（代码审查）。
- bug（缺陷）排查。
- 上线风险评估。
- ES（Elasticsearch）查询检查。
- MQ consumer（消息队列消费者）检查。

Skills（技能）的价值是把经验流程固定下来，降低每次临时 prompt（提示词）的不稳定性。

### Q27：Multi-Agent（多智能体）会不会更快？

在独立子任务上会更快，例如多个模块并行阅读。  
但如果多个 agent（智能体）共享状态或同时改同一文件，可能更混乱。使用 Multi-Agent（多智能体）要明确：

- 任务是否独立。
- 输出如何汇总。
- 谁负责最终判断。
- 是否允许写文件。

### Q28：如何做 Agent Evaluation（智能体评估）？

可以从四类指标开始：

- 正确性：测试是否通过，需求是否满足。
- 安全性：是否越权，是否泄露敏感信息。
- 可维护性：diff（差异）是否小，是否符合架构。
- 成本体验：耗时、token（令牌 / 模型计费与长度单位）、工具调用次数、人工打断次数。

## H. 组织与流程

### Q29：如何把 Codex 引入团队流程？

建议分阶段：

1. 个人试用：整理经验和失败模式。
2. 仓库规则：补齐 AGENTS.md（项目级智能体规则文件）。
3. 小任务试点：限定低风险场景。
4. 审计流程：统一输出格式和验证要求。
5. 工具扩展：按需建设 MCP / Skills（模型上下文协议 / 技能）。
6. 评估复盘：统计成功率、失败原因和风险事件。

### Q30：团队应该禁止哪些用法？

建议禁止或强审批：

- 未审查直接合并 agent（智能体）改动。
- 让 agent（智能体）直接操作生产数据。
- 在 prompt（提示词）或 AGENTS.md（项目级智能体规则文件）写 secrets（密钥 / 敏感凭据）。
- 高风险业务动作使用 full-auto（全自动模式）。
- 未验证就声称完成。
- MCP（模型上下文协议）工具返回敏感原文。

### Q31：如何说服团队这不是“玩具”？

不要用炫技 demo（演示）说服。用工程证据说服：

- 固定一类小任务。
- 记录前后耗时。
- 记录缺陷率。
- 记录测试通过率。
- 复盘失败案例。
- 形成可复制流程。

### Q32：如何防止团队过度依赖 Codex？

把 Codex 定位为协作工具，而非最终责任人。要求：

- 人必须 review（审查）。
- 高风险必须审批。
- 结果必须验证。
- 业务语义由人确认。
- agent（智能体）输出不能跳过工程流程。

# 讲师收尾建议

如果时间允许，最后用这段话收尾：

> 我们真正要学的不是某个按钮怎么点，而是如何把 AI agent（AI 智能体）放进工程体系里。Codex 的能力来自模型，但可信度来自运行时边界：上下文让它少猜，工具让它能做事，权限让它不乱做，记忆让经验可复用，验证让结果可信。团队落地时，不要追求一次性全自动，而要从小任务、小权限、小闭环开始。

# 分享后可布置的行动项

1. 为当前团队最常用仓库写第一版 `AGENTS.md`（项目级智能体规则文件）。
2. 选 3 类低风险任务做 Codex 试点。
3. 建立 agent（智能体）输出验收模板。
4. 梳理哪些内部系统适合 MCP（模型上下文协议）接入，哪些必须禁止。
5. 每月 review（审查）一次 Memories（记忆条目）或实践经验，把稳定规则沉淀到团队文档。
