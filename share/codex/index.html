OpenAI Codex / Agent Runtime Deep Dive

Codex Agent Runtime：架构、记忆与扩展机制

面向团队内部分享的中文友好版文档。它把 Codex 解释为一个可审计、可治理、可恢复、可扩展的工程执行系统，而不只是一个会写代码的聊天助手。

Agent Loop Context Engineering AGENTS.md Memories MCP / Skills / Plugins

文档摘要

01 / Summary

Codex 的核心不是“模型会写代码”，而是“模型在工程边界内循环执行任务”。它通过 Agent Loop 把自然语言需求转成工具调用、文件修改、命令执行、测试验证和最终交付。

理解 Codex 时，应把它看成一套 coding agent runtime：模型负责推理，运行时负责上下文、工具、安全、记忆、审计和恢复。

30 秒结论：Codex = Agent Loop + Prompt/Context 构造 + Tool Runtime + Sandbox/Approval + Thread/Rollout + AGENTS.md/Memories + MCP/Skills/Plugins。
给团队的定位 不要把 Codex 当成“自动程序员”

更稳妥的接入方式是把它当成“带权限边界的工程执行系统”：让 agent 提出动作，让运行时限制动作，让人审批高风险动作，让审计记录全过程。

适用场景

02 / Scenarios
Local CLI 本地仓库协作

读取项目规则、检索代码、修改文件、运行测试、解释差异，适合日常开发闭环。

Cloud Agent 长任务与并行任务

适合异步实现、批量修复、生成 PR、代码审查和需要等待结果的工程任务。

App Server IDE 与内部平台接入

通过协议管理 thread、turn、event，让不同界面复用同一套 agent runtime。

最适合的任务形态：目标明确、验收标准清楚、权限边界可控、能通过测试或命令验证的工程任务。

核心概念

03 / Concepts
Agent能感知上下文、自主决策、调用工具并循环执行直到完成目标的系统。
Agent Loop构建上下文、模型推理、工具调用、结果写回、再次推理的闭环。
Turn从一次用户输入到 agent 返回最终消息的完整执行周期。
Tool Call模型提出的结构化外部动作，例如检索文件、执行命令、应用补丁。
Harness连接模型、工具、上下文、权限、状态的运行框架，是 Codex 的共享核心。
Prompt发送给模型的完整输入，包含系统指令、工具定义、历史和当前用户消息。
Context Window单次推理可容纳的 token 上限，需要通过摘要、截断和任务拆分管理。
Prompt Caching服务端复用相同前缀的计算结果，关键是保持静态前缀稳定。
Rollout一次 agent 会话的完整执行轨迹，可用于恢复、复盘和记忆提炼。
ZDRZero Data Retention，零数据留存。Codex 的无状态请求设计服务于企业合规。

Agent Loop 单轮流程

agent-loop.txt
用户输入
  -> 构造 Prompt：system + tools + instructions + history + current user message
  -> POST /v1/responses
  -> 解析模型输出
     -> 如果是 tool call：执行工具，追加 tool result，继续推理
     -> 如果是 assistant message：结束本轮 turn，交还控制权
关键理解：模型只提出下一步意图，真实动作由运行时执行。工具路由、审批、沙箱、错误处理和结果记录都不应该交给提示词“自觉完成”。

记忆机制

05 / Memory

Codex 的记忆机制可以理解为双层架构：AGENTS.md 是人工维护的静态规则层，Memories 是 agent 从历史会话中自动提炼的动态经验层。

Layer 1 AGENTS.md 静态指令层

适合放稳定规则：测试命令、代码风格、架构边界、禁改区域、权限要求。项目级文件可提交 Git，团队共享。

Layer 2 Memories 自动生成层

适合沉淀会话中涌现的知识：bug 模式、临时决策、偏好和复盘结论。通常存储在用户本地记忆目录。

加载顺序

config.toml读取全局配置、模型、权限和 memories 开关。
Global AGENTS读取用户级 ~/.codex/AGENTS.md 或 override。
Project AGENTS从 repo root 到 cwd 逐层收集项目规则。
Memories按配置读取自动记忆摘要。
Prompt合并为 developer 指令，注入本次上下文。
记忆类型 维护方式 适用内容 团队共享
AGENTS.md 人工编写 稳定规范、测试命令、架构边界、安全约束 可提交 Git，共享
AGENTS.override.md 人工编写 临时或子目录强制覆盖规则 谨慎共享，需说明原因
Memories 自动提炼 历史会话经验、偏好、复盘知识 默认个人本地，不自动共享
工程建议：让 Memories 自动积累经验，但定期把稳定、可复用、对团队有价值的内容提升到项目 AGENTS.md。

扩展机制

06 / Extensions
MCP 把外部系统变成工具

通过 Model Context Protocol 接入数据库、知识库、内部 API、发布平台。工具必须有参数约束、权限边界和审计记录。

Skills 把经验固化为工作流

适合沉淀专项流程，例如代码审查、系统排障、文档生成、表格处理、演示文稿制作。

Plugins 组合技能、工具与应用

适合把团队内部工具链封装为可安装能力，降低重复配置和接入成本。

Multi-Agent 并行处理独立子任务

适合大规模代码阅读、模块化排查、竞品/资料调研。要控制共享状态，避免多个 agent 同时写同一文件。

CI/CD 交付链路自动化

可把构建、测试、发布、回滚检查接入 agent，但生产写操作必须使用最小权限和审批链路。

MCP 安全边界：Codex 对原生 shell 工具有 sandbox 和 approval，第三方 MCP 工具需要 MCP Server 自己实现权限控制、参数校验和敏感数据保护。

示例代码

08 / Examples

AGENTS.md 模板

AGENTS.md
# 项目协作规范

## 语言
- 用户可见说明使用简体中文。
- 代码、命令、API 路径、配置 key 保持英文。

## 修改规则
- 先阅读相关代码和测试，再做最小安全修改。
- 不做无关重构，不重排无关文件。
- 不新增依赖，除非明确说明必要性和风险。

## 验证
- 修改后优先运行最相关测试。
- 如果测试未运行，必须说明原因。
- 不要声称测试通过，除非实际运行且退出码为 0。

## 安全
- 不打印 secrets、tokens、cookies、authorization headers。
- 支付、订单、库存、权限、生产写操作必须显式说明风险。

Agent Loop 伪代码

agent-loop.rs
loop {
    let context = build_context(thread, project_rules, tools, memories);
    let response = model.sample(context).await?;

    if response.is_assistant_message() {
        thread.record(response).await?;
        break;
    }

    let calls = tool_router.parse(response)?;
    let results = orchestrator.execute_with_policy(calls).await?;
    thread.record(results).await?;
}

最佳实践

09 / Best Practices
  • 为每个仓库维护清晰的 AGENTS.md，写明构建命令、测试命令、架构边界、禁改区域和审批要求。
  • 把任务拆小：一次只交付一个可验证目标，避免上下文漂移和大范围误改。
  • 提前声明验收标准：需要跑哪些测试、检查哪些页面、查看哪些日志。
  • 工具最小权限：读写分离、资源范围限制、参数校验、敏感字段脱敏。
  • 高风险动作显式审批：生产写操作、支付、库存、订单状态、权限变更都不能静默执行。
  • 保留执行证据：transcript、命令输出、测试结果、审批记录和最终 diff 都应该可追溯。
  • 定期清理和提升记忆：把 Memories 中稳定可靠的团队共识迁移到 AGENTS.md。
  • 为扩展工具写清楚失败处理：timeout、retry、幂等、重复请求、权限不足和审计日志。

常见误区

10 / Mistakes
误区 1：把安全写进提示词就够了

提示词不是安全边界。真实工程动作必须由 approval、sandbox、permission policy 和 audit log 约束。

误区 2：让 agent 一次做完整个大项目

大任务会放大上下文污染、错误累积和验证困难。更可靠的方式是小任务、小闭环、持续验证。

误区 3：Memories 可以替代 AGENTS.md

Memories 更像个人经验沉淀，不适合作为团队规范来源。稳定规则应写入项目 AGENTS.md 并提交 Git。

误区 4：MCP 工具越多越好

工具越多，权限、审计、缓存命中和模型选择成本越高。每个工具都应该有明确用途和最小权限。

误区 5：只看最终答案，不看执行证据

Agent 的可信度来自过程证据：它看了什么、改了什么、跑了什么、哪里失败、最后如何验证。

可复用提示词

11 / Prompt

下面是对原始提示词的优化版。它更明确地限定了产物形态、内容结构、视觉约束和输出边界，适合后续生成同类单文件技术文档。

single-file-tech-doc-prompt.md
你是一名资深前端工程师、技术文档设计师和技术分享讲师。

请基于我提供的主题和内容，生成一个「可直接本地打开、可分享、可复制到飞书/语雀/内部 wiki 的单文件 HTML 技术文档」。

硬性要求：
- 只输出完整 HTML 代码，不要额外解释。
- 单文件 HTML，内联 CSS，原生 JavaScript。
- 不依赖任何外部资源、CDN、图片、字体或框架。
- 兼容 Chrome / Edge / Safari。
- 响应式布局，移动端可读。
- 支持浏览器打印为 PDF。
- 技术名词、代码、命令、API 路径、配置 key 保持英文。
- 用户可见说明使用简体中文。

内容目标：
- 信息层次清晰，适合技术分享和团队内部传播。
- 先给结论，再展开细节。
- 对复杂概念提供类比、流程、表格和代码示例。
- 对高风险或易误解内容使用 callout 提醒。
- 不编造事实；不确定内容标注为「需要验证」。

页面模块：
1. Hero 标题区：主题、定位、一句话摘要、关键词标签。
2. 文档摘要：30 秒结论和适合谁读。
3. 适用场景：说明什么时候应该使用该方案。
4. 核心概念：用短定义解释关键术语。
5. 架构/流程说明：用 CSS 卡片或流程图表达主流程。
6. 表格对比：至少一张对比表，突出取舍。
7. 示例代码：提供可复制代码块。
8. 最佳实践：可执行的团队落地建议。
9. 常见误区：列出错误理解和修正建议。
10. 总结：3 到 5 条可带走的结论。
11. 可选交互：目录高亮、返回顶部按钮、代码复制按钮。

视觉要求：
- 使用现代卡片式布局，但不要过度装饰。
- 字体大小舒适，正文行距 1.6 到 1.8。
- 颜色克制，主色不超过 2 个，强调色不超过 2 个。
- 重点内容用 callout 样式突出。
- 表格边界清晰，移动端可横向滚动。
- 代码块使用深色背景，带标题和复制按钮。
- 打印模式隐藏目录、按钮等交互元素。

文档主题：
【填写主题】

内容素材：
【粘贴 Markdown、笔记、代码片段或大纲】

补充要求：
【填写是否需要加入扩展章节、团队落地建议、FAQ、对比表等】

总结

12 / Closing

Codex 的价值不在于替代工程师，而在于把模型能力放进可控的工程运行时。

团队真正需要建设的是上下文、权限、工具、记忆和验证体系。上下文让 agent 少猜，权限让动作受控，工具让执行落地，记忆让经验复用，验证让交付可信。

落地时请坚持一个原则：放大上下文，缩小动作半径；让 agent 做可验证的小闭环，而不是无边界的大承诺。
