04 核心引擎

Agentic Loop 主循环

函数签名

// 1729行心脏 — Async Generator export async function* query( params: QueryParams ): AsyncGenerator< StreamEvent | Message, Terminal // return: 终止原因 > { let state: State = initState(params) while (true) { // 1. Pre-compaction // 2. API call + stream // 3. Tool authorization // 4. Execute tools // 5. Recovery // 6. Stop check state = applyTransition(state) } }
Java 类比：Flux<T>（Project Reactor），但用语言原生语法，无需框架依赖，错误通过 throw 传播，终止原因从 return 拿。

不可变参数 vs 可变状态

// 不可变 (整个循环不变) type QueryParams = { systemPrompt: string canUseTool: CanUseToolFn maxTurns?: number } // 可变 (每次迭代原子替换) type State = { messages: Message[] turnCount: number transition: Continue | undefined } // ✅ 原子替换，防止中间状态 state = { ...state, messages: newMessages, turnCount: state.turnCount + 1, transition: { reason: 'next_turn' } }
来自 React setState 哲学：状态整体原子替换，不逐字段修改，确保任何时刻 state 对象都是一致快照。

06 工具系统

Tool System — 50+ 工具设计哲学

核心原则：每个工具自带 input schema、permission level、execution logic。没有工具间共享的可变状态——BashTool 和 FileReadTool 在同一注册表，但风险画像完全不同，架构分别对待。
FileReadTool / GlobTool / GrepTool
只读 · isConcurrencySafe: true
流式响应时就开始并行执行，不等模型停止生成
FileEditTool / FileWriteTool
写操作 · isConcurrencySafe: false
独占执行，不与任何其他工具并发
BashTool / AgentTool
高危 · gated + sandboxed
4级授权管线 + 沙箱隔离 + 子 Agent 隔离 context
private canExecuteTool(isConcurrencySafe: boolean): boolean { const executing = this.tools.filter(t => t.status === 'executing') return ( executing.length === 0 || (isConcurrencySafe && executing.every(t => t.isConcurrencySafe)) ) // Java 类比：ReadWriteLock — 多读一写 }
07 Context 管理

Context 压缩管线

核心瓶颈：LLM Context Window 是有限资源。所有优化都围绕这个约束展开。5级策略按 cheapest-first 顺序执行——一旦当前层达到阈值，不触发更高成本的层。
L1
Tool Result Budget
截断过大工具结果（文件读取、搜索结果），保留关键内容
L2
Snip Compact HISTORY_SNIP flag
直接丢弃最旧消息块，最激进但成本最低
L3
Microcompact
Prompt Cache 感知，选择性清除单个工具结果，不破坏缓存边界
L4
Context Collapse CONTEXT_COLLAPSE flag
分阶段折叠消息块，生成压缩视图，不修改原始数据
L5
Auto-Compact（LLM 摘要）
阈值触发，调用 LLM 对整段对话摘要——最贵，L4 通过后跳过
Java GC 类比：Minor GC (L1-L2) → CMS incremental (L3-L4) → Full GC (L5)。Context Collapse 必须在 Auto-Compact 之前——Collapse 成功则跳过一次 API 调用费用。

09 权限系统

Permission System — Defense in Depth

工具调用授权管线（4级）

①
Pre-filter
工具名/类型快速过滤
→
②
PreToolUse Hook
用户自定义脚本拦截
→
③
Rule Eval
allowList/denyList 路径检查
→
④
Permission Handler
ML 分类器 or 人工确认
Default Mode
不确定操作均询问用户，最高控制权，适合敏感操作环境
Auto Mode
ML 分类器评估风险，低风险自动执行，高风险仍需确认
Headless / CI Mode
无人值守，依赖预设规则集，适合 CI/CD 流水线
Java 类比：Spring Security SecurityFilterChain + 自定义 AccessDecisionVoter，但鉴权对象是"AI 即将执行的动作"而非"HTTP 请求"。Anthropic 信念：人类始终保留对 Agent 目标追求方式的控制权。

11 多 Agent

Multi-Agent 架构

子 Agent 隔离模型

  • 独立 file cache（从父 Agent 克隆）
  • 独立 AbortController（异步）或共享（同步）
  • 独立 transcript JSONL sidechain
  • 过滤后的工具池（按 Agent 定义限制）
  • 以纯文本向父 Agent 返回结果
  • 权限不继承，resume 时不恢复子 Agent 权限
Java 类比：ForkJoinTask，但 worker 是独立 LLM 实例，通信通道是自然语言而非方法调用返回值。
未发布 Coordinator Mode：feature flag 已存在，多 worker 并行，类似 MapReduce Map 阶段。

13 扩展架构

四种扩展机制

扩展机制Java/Spring 类比作用层次典型用途
MCP ServerSPI + JSON-RPC进程级隔离数据库、Git、自定义工具注册
Hooks@Aspect / AOP 切面工具生命周期审计日志、参数验证、安全扫描
Skills@Configuration 模块指令/知识项目规范注入、领域知识复用
PluginsMaven Plugin能力包集成 Jira、Slack、第三方服务
{ "type": "web_search_20250305", "name": "web_search", "input_schema": { "type": "object", "properties": { "query": { "type": "string" } } } } // Claude Code 作为 MCP Client，每个 Server 独立进程（进程级隔离）
为什么 4 种机制而非 1 种？MCP 扩展能力边界，Hooks 扩展控制边界，Skills 扩展知识边界，Plugins 扩展集成边界。强行统一会让每种场景都用不合适的抽象。

① 分级分层（Graduated Layering）：每个关切（安全、权限、压缩、扩展）都有独立分级机制，失败模式被隔离，不同级别可独立演进。
② Append-Only 偏好可审计性：Session transcript 是 JSONL append-only 日志，支持 fork/resume/审计。类比 Event Sourcing，以查询能力换可观测性。
③ 确定性 Harness 内使用模型判断：不构建显式规划器，投资稳定基础设施（context 管理、工具路由、错误恢复），让 LLM 在确定性 harness 内自主决策。
数据佐证：Anthropic 内部调研 132 名工程师，约 27% 的 Claude Code 辅助任务是"没有这个工具就不会尝试的工作"——实现质的工作流创新，而非仅仅加速现有流程。
