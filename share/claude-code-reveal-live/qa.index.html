Company Internal AI Agent Q&A（公司内部 AI Agent 讨论）

Claude Code / AI Agent内部接入现场讨论稿

假设这些问题是围绕“公司内部接入 Claude Code / AI Agent 架构”做现场讨论，建议先对齐问题边界，再进入回答。核心不是让 Agent 直接接管业务，而是把它放进权限、审批、审计和可观测体系里。

先对齐 5 个问题

01
MCP Server 怎么做权限隔离？讨论工具能力、资源范围、用户身份和审计边界。
02
订单、支付、库存这类高风险业务，哪些动作绝不能自动执行？讨论真实业务后果和审批底线。
03
Agent 失败时，怎么做可观测性和问题追踪？讨论 trace、transcript、tool 调用链和恢复能力。
04
长期记忆应该放在文件、数据库，还是知识库？讨论不同记忆类型的存储分层。
05
多人协作时，谁来审批 Agent 的工具调用？讨论审批人如何由动作风险和业务归属决定。

逐题回答

01

MCP Server 怎么做权限隔离？

按“用户身份 + 工具能力 + 资源范围 + 操作级别”隔离。

  • MCP Server 不应该拥有全局超级权限，应继承调用人的身份，或绑定最小权限服务账号。
  • 每个 tool 明确定义可访问资源，例如只能查订单、不能改订单。
  • 读写分离：查询类 tool、变更类 tool、危险操作 tool 分开注册。
  • 高风险 tool 必须二次审批，例如退款、改库存、发券、改支付状态。
  • 所有调用记录审计：调用人、Agent、tool、参数摘要、结果、耗时、审批人。
  • 敏感字段脱敏，禁止把 token、手机号、身份证、银行卡等原样返回给模型。
02

高风险业务，哪些动作绝不能自动执行？

不能让 Agent 无审批地直接执行会产生真实业务后果的动作。

修改订单 / 支付 / 履约状态 退款 / 补偿 / 打款 / 扣款 释放 / 锁定 / 扣减库存 修改票价 / 结算金额 / 账户余额 发送正式通知 / 短信 / 邮件 批量更新生产数据 跳过风控 / 审批 / 幂等 / 状态机校验 生产 SQL 写操作 / 删除操作

Agent 可以辅助“分析、生成方案、生成待审批变更单”，但最终执行应走已有业务系统、权限、审批和审计链路。

03

Agent 失败时，怎么做可观测性和问题追踪？

核心是把 Agent 当成一个分布式执行系统来观测。

  • 记录每轮模型输入、输出摘要。
  • 记录 tool 调用链路、参数摘要、返回结果、异常堆栈。
  • 记录当前任务 ID、用户 ID、会话 ID、traceId。
  • 记录权限审批、文件变更、命令执行、外部 API 调用。
  • 标记失败发生在哪一步：理解需求、规划、调用工具、执行命令、验证结果。
  • 支持从 transcript 重放关键步骤。
  • 长任务保存 plan、progress、checkpoint。
  • 失败后区分模型判断错误、工具错误、权限错误、业务校验错误还是外部系统错误。
04

长期记忆应该放在哪里？

看记忆类型，不建议只选一种。

  • 文件：适合项目规则、开发约定、架构说明，例如 AGENTS.md、设计文档、运行手册。
  • 数据库：适合结构化、可审计、可权限控制的数据，例如任务状态、审批记录、用户偏好。
  • 知识库 / 向量库：适合大量非结构化内容检索，例如历史方案、FAQ、事故复盘、代码解释文档。
  • 实践分层：稳定规则放文件，强一致业务数据放数据库，大量参考资料放知识库，当前任务上下文放 transcript / checkpoint。
05

多人协作时，谁来审批 Agent 的工具调用？

审批人应该由“动作风险”和“业务归属”决定，而不是固定一个人。

  • 普通读操作：通常不需要审批，但要审计。
  • 代码改动：模块 owner 或 reviewer 审批。
  • 生产查询：系统 owner 或值班人员审批。
  • 生产写操作：业务 owner + 技术 owner 双审批。
  • 支付、退款、库存、订单状态：必须走现有业务审批流。
  • 批量操作：需要更高级别审批，并要求预览影响范围和回滚方案。
一句话总结

Agent 可以提出动作，系统负责限制动作，业务负责人审批高风险动作，审计系统记录全过程。
