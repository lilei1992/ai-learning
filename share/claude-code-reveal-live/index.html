00 / 先建立直觉

先用一句话讲清楚：Claude Code 是什么？

一个会读项目、改文件、跑命令、看结果、再继续调整的命令行 AI 编程助手。
普通聊天机器人只会“回答”；Claude Code 的核心是“边想边行动”。

聊天式 AI

用户提问，模型回答。执行动作通常要靠人复制粘贴。

Agent 式 AI

模型可以选择工具，执行动作，读取反馈，再决定下一步。

小白第一关是理解 Agent 和聊天机器人的差异。强调“行动闭环”，不要先讲技术栈。
00.5 / Learning Model（学习模型）

这次学习可以先记住一个类比

AI Agent像一个实习工程师：能理解目标，也会自己尝试下一步。
Tools（工具）像电脑上的能力：读文件、改文件、跑命令、访问外部系统。
Permission（权限）像审批制度：低风险自动做，高风险必须问人。
Context（上下文）像桌面资料：放太多会乱，必须归档、压缩和总结。

你不需要先懂源码。只要理解“人给目标、模型做判断、工具去执行、权限控边界、上下文保记忆”这条主线。

02 / Learning Path（学习路径）

今天只抓住 5 个问题

  1. 它为什么不是普通 CLI？因为它有持续行动的 Agent Loop（Agent 循环）。
  2. 它怎么决定下一步？模型判断 + 确定性基础设施。
  3. 它怎么安全地改代码？Permission（权限）、Rule（规则）、Human Confirmation（人工确认）、Sandbox（沙箱）。
  4. 它怎么避免上下文爆掉？Context Compaction（上下文压缩）和 Summary（摘要）策略。
  5. 它怎么接入外部世界？MCP、Hooks（钩子）、Skills（技能）、Plugins（插件）。
03 / Glossary（术语表）

先把常见英文概念翻译成人话

Agent会自主选择下一步动作的 AI 执行体
Loop循环：观察、思考、行动、反馈、再行动
Context上下文：模型当前能看到的全部信息
Compaction压缩：把冗长上下文变短但尽量不丢重点
Permission权限：哪些动作能自动做，哪些必须问人
Sandbox沙箱：限制危险命令影响范围的隔离环境
Transcript会话记录：每一步输入输出的流水账
Harness围栏：让模型自由发挥但不能越界的基础设施

05 / 总览

五层架构：先看地图，再看细节

表现层CLI、终端 UI、SDK Server、远程入口
核心层Agent Loop（Agent 循环）、Streaming（流式调用）、State Transition（状态转移）
安全层Permission System（权限系统）、Hook（钩子）、Rule Evaluation（规则评估）、Sandbox（沙箱）
状态层Context Assembly（上下文组装）、Transcript（会话记录）、Memory（记忆）、Compaction（压缩）
后端层Claude API、MCP 服务、文件系统、Shell、网络

核心思想：让 LLM 负责判断“下一步做什么”，让工程系统负责“能不能做、怎么安全做、做完如何记录”。

07 / Inside Agent Loop（Agent 循环内部）

一轮 Turn（回合）里发生了什么？

1. Pre-Request Compaction（请求前压缩）先看上下文是否太长
2. API Streaming（API 流式调用）模型边生成边返回事件
3. Tool Authorization（工具授权）判断工具能否执行
4. Tool Execution（工具执行）只读并发，写操作串行
5. Recovery（错误恢复）超限、失败、备用模型
6. Stop Check（停止判断）结束或进入下一轮
while (notDone) {
  context = compactIfNeeded(context)
  response = callLLM(context)
  actions = authorize(response.tools)
  result = runTools(actions)
  context = append(result)
}
不用先看懂代码

它表达的是一个学习重点：Agent 每一轮都会先整理资料，再让模型判断，然后检查权限，最后执行工具并把结果写回记忆。

08 / 工具系统

工具是 Agent 的手脚，但每只手风险不同

Read / Glob / Grep只读工具，可并发执行
Edit / Write会改文件，需要独占执行
Bash能执行命令，需要授权和沙箱
MCP Tool（MCP 工具）外部服务动态提供，按服务授权

学习重点：工具越能改变真实世界，越需要权限、沙箱和记录。只读工具和写入工具不能用同一套规则。

09 / Live Demo（实时演示） B

只读工具可以并发，写操作必须排队

理解方式：只读动作像“同时看几份资料”；写入动作像“真正动手改东西”，必须一个个确认和执行。

10 / Context（上下文）

Context Window（上下文窗口）：模型工作台有限

人类工作台

桌面太乱时，你会把旧资料归档，只留下当前任务相关内容。

LLM 工作台

上下文太长时，系统要截断、折叠或总结，否则请求会超限或成本变高。

需求代码片段工具结果错误日志历史对话项目规则

Context Window 直译是“上下文窗口”，可以理解为模型一次能看见的资料上限。

12 / Permission System（权限系统）

Permission System（权限系统）：行动边界

Pre-filtering（预过滤）明显禁止的工具直接挡掉
Hook（钩子）用户脚本可以检查或改参数
Rule Evaluation（规则评估）匹配 allow / deny 规则
Human Confirmation（人工确认）不确定或高风险动作问用户
学习重点：Agent 的价值来自自主性，但人类必须保留对行动边界的控制。

16 / 扩展机制

为什么有 4 种扩展机制，而不是全靠 MCP？

MCP扩展 Capability Boundary（能力边界）：多一个外部工具
Hooks（钩子）扩展 Control Boundary（控制边界）：执行前后插入检查
Skills（技能）扩展 Knowledge Boundary（知识边界）：注入任务方法和领域规范
Plugins（插件）扩展 Integration Boundary（集成边界）：打包技能、工具、连接器

不同机制解决不同层级的问题。强行统一，反而会让每种场景都很别扭。

17 / 设计哲学

这套架构真正值得学的是边界感

Human Authority（人类决策权威）目标和高风险动作仍由人控制。
Safety & Privacy（安全与隐私）权限、沙箱、日志遮蔽缺一不可。
Reliable Execution（可靠执行）工具失败、超限、重试都要可恢复。
Capability Amplification（能力放大）让模型能调用工具，而不是只会生成文本。
Context Adaptation（上下文适应）长任务必须管理记忆和历史。
18 / 给 Java 开发者

把 Claude Code 映射到熟悉的后端概念

Claude Code 概念Java / Spring 类比本质差异
Agent Loop（Agent 循环）Reactor Flux驱动源是模型输出，不是普通数据流。
Context Compaction（上下文压缩）分级 GC“内存”是 token，成本是 API 费用。
Permission Pipeline（权限管线）SecurityFilterChain鉴权对象是 AI 即将执行的动作。
Concurrency Safety（工具并发安全）ReadWriteLock这是调度策略，不是 JVM 锁。
Transcript（会话记录）Event Sourcing记录的是人、模型、工具交互全过程。
SubAgent（子 Agent）ForkJoinTaskWorker 是独立 LLM 实例。
19 / 小白总结

如果只记住 4 句话

Agent 不是“更会聊天”，而是“能闭环行动”。
LLM 做判断，工程系统做边界和恢复。
工具越强，权限、沙箱和审计越重要。
长任务的关键不是 Prompt（提示词），而是 Context Management（上下文管理）。
20 / 现场讨论

可以继续展开的 5 个问题

  • 如果接入公司内部系统，MCP Server 应该怎么做权限隔离？
  • 订单、支付、库存这类高风险业务，哪些动作绝不能自动执行？
  • Agent 失败时，怎么做可观测性和问题追踪？
  • 长期记忆应该放在文件、数据库，还是专门的知识库？
  • 多人协作时，谁来审批 Agent 的工具调用？

Feature Flag（功能开关）：未来能力线索

6 个开放研究方向（学术论文式总结）

静默失败与可观测性缺口Agent 失败时往往无声，需要更好的追踪和评估框架。
记忆作为一等公民跨 session 的长期记忆不能只依赖 CLAUDE.md，需要系统级方案。
Harness 边界演化在哪里、何时、做什么、与谁协作，边界和置信任度会动态调整。
架构解耦推理、执行、会话三层进一步分离，允许独立扩展和替换。
治理与监督随 Agent 自主性增强，需要在规模上维持人类监督。
主动架构从响应式（等待输入）转向主动式（主动感知、规划、行动）。

注意：这些更适合作为架构观察和研究线索，不应解读为官方路线图或确定承诺。
