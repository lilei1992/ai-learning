Context Engineering

为什么优秀工程师的 Codex Token 消耗只有普通人的 1/10

让团队学会按需动态组装上下文

01 Token 为什么爆炸

问题不是模型不够强，而是上下文太脏

  • 把整个项目源码喂给模型
  • 携带大量无关历史对话
  • 粘贴几十 MB 日志
  • 需求只改 10 行，输入却几十万 Token
错误做法

项目全量喂给模型

src/ ├── controller/ ├── service/ ├── dao/ ├── mq/ ├── es/ ├── payment/ ├── user/ └── 5000+ files

模型会读，但不代表这些信息都有价值。

正确做法

按需动态组装上下文

当前任务 + 当前文件 + 调用链相关代码 + 业务规则摘要 + 必要历史结论
02 Context Window

模型看到的不是项目，而是一个超长文本

Codex 不会天然理解你的仓库、目录、历史背景。

它真正能处理的，是你塞进 Context Window 的内容。

Garbage In，Garbage Out。

核心公式

Context 应该只包含必要信号

Context = 当前任务 + 相关代码 + 相关文档 + 必要历史
不要包含

这些都是 Token 噪音

无关代码

不在调用链、不影响本次变更的代码。

无关日志

重复 INFO、DEBUG、历史运行日志。

无关对话

讨论过程、废弃方案、闲聊记录。

无关文档

整篇设计稿、过期说明、背景材料。

03 成本分析

真正贵的是 Input Token

普通方式

120k

Input Tokens

优秀方式

Input Tokens

输出代码可能差不多，但成本相差 10 倍以上。

04 六个降本技巧

让 Token 花在刀刃上

  • 只提供相关代码
  • 按需动态检索，不预加载全部内容
  • 压缩历史，只保留结论
  • 裁剪日志，只保留错误、堆栈、关键参数
  • 文档摘要化，只保留规则和约束
  • 建设团队项目知识库和 AGENTS.md
技巧示例

日志应该这样喂

错误

50MB full.log

正确

Exception StackTrace Request ID 关键入参 最近一次变更
05 真实项目案例

新增订单超时状态

普通工程师

  • 订单模块全部代码
  • MQ / ES / 支付 / 用户全部带入
  • 输入约 120000 Token

优秀工程师

  • OrderStatusEnum
  • OrderService 相关方法
  • 相关单测
  • 需求和验收标准
本质差异

不是 Prompt 能力，而是 Context Engineering 能力

普通工程师在优化 Prompt。

优秀工程师在优化 Context。

06 团队规范

团队统一上下文输入标准

每次请求 Codex 前，必须说明： 1. 当前任务 2. 相关代码范围 3. 相关业务规则 4. 不允许修改的边界 5. 期望输出格式
AGENTS.md 模板

把团队规范写进项目根目录

# AGENTS.md ## Context Rules Context = 当前任务 + 相关代码 + 相关文档 + 必要历史 禁止加入： - 无关代码 - 无关日志 - 无关文档 - 无关历史对话 ## Code Change Rules - 最小修改 - 向后兼容 - 保持现有风格 - 禁止无关重构
Final

Less Tokens · More Accuracy
