Token 爆炸

Token 为什么爆炸

绝大多数 token 浪费来自同一个根因

普通工程师的做法
  • 把整个项目目录粘进去
  • 每次对话带上完整历史记录
  • 包含无关模块的代码
  • 把 build 日志全量输出
  • 用完不清理 context
优秀工程师的做法
  • 只给当前任务相关的 2-3 个文件
  • 必要时主动 /compact 压缩
  • 用 CodeGraph 精准定位符号
  • grep 过滤后再传日志
  • 任务结束即开新 session

Context 原理

Context Window 工作原理

模型每次推理只能看到窗口内的内容，多一个 token 都是钱

Context Window（以 200K 为例）
System / AGENTS.md
固定开销，每轮必付
相关代码文件
真正有用的部分
历史对话噪音
旧轮次的工具输出、重复信息
剩余可用
实际能放新内容的空间

输入 token

System prompt + 历史对话 + 当前消息 + 工具调用结果，全部算钱。大模型看到的每一个字都有成本。

输出 token

模型生成的每一个 token 通常比输入贵 3-6 倍。让模型少废话、关掉不必要的 extended thinking，立竿见影。

降本技巧

6 个降本技巧

按收益从高到低排序，每条都有实测数据

  • 1
    精简 AGENTS.md — 节省 30–90% 3847 token 的 AGENTS.md 精简到 312 token，质量无回归。只写项目"形状"，细节按路径引用不预加载。
  • 2
    CodeGraph MCP — 减少 70% 工具调用 用预索引知识图谱替代 grep 探索。Agent 发现阶段从 20-50 次工具调用降到 3-6 次，token 节省 35%。
  • 3
    RTK / Headroom 拦截 shell 输出 — 节省 60–95% mvn test、git status 的原始输出在进入 LLM 前被压缩为纯信号，常见命令节省 60-90%。
  • 4
    模型分级路由 — 节省 40–60% 探索、日志、文档查找用 gpt-5.4-mini，主线推理用 gpt-5.5。complex thinking 只给架构级问题开。
  • 5
    精确 Prompt + 一任务一 session — 节省 20–50% "修复 src/auth.ts 第42行的 bug" 比 "认证有问题帮我查" 消耗少得多。任务完成即 /clear 或开新会话。
  • 6
    Prompt Caching — 缓存命中折扣 75% 把稳定内容放在 AGENTS.md 最前面，让缓存命中最大化。重复内容只收 25% 的价格。
综合使用后实测：整体 token 消耗降低 90%+

真实案例

真实项目案例对比

同一个 Spring Boot 服务重构任务，两种做法的差异

错误：全量喂入
  • 把整个 src/ 目录读入
  • 粘贴完整 maven build 日志
  • 带上所有历史对话
  • AGENTS.md 超过 3000 token
  • 每次都重新描述项目背景
正确：按需动态组装
  • codegraph_search 定位相关类
  • 只传 grep 过滤后的报错行
  • /compact 后再继续
  • AGENTS.md 控制在 300 token
  • 路径引用规范文档，不预加载
正确的 Context 组装公式
Context = 当前任务 + 相关代码 + 相关文档 + 必要历史
无关代码 无关日志 无关对话 ← 不要包含

团队规范

团队 AGENTS.md 模板

复制即用，控制在 300 token 以内

AGENTS.md — Spring Boot 团队模板
## 项目 框架: Spring Boot 3.x / Java 21 / Maven 包结构: controller / service / repository / domain / config 测试命令: mvn test -pl <module-name> 构建命令: mvn clean install -DskipTests ## 使用 CodeGraph（必须） - 探索代码库 → codegraph_search 优先于 grep - 修改接口前 → codegraph_impact 检查影响范围 - 追踪调用链 → codegraph_callers / callees ## Context 规范 - 每个任务只读相关文件，不全量扫描 - shell 输出超过 100 行时先 grep 过滤 - 任务完成后执行 /compact 或开新 session - 详细规范见 ./docs/conventions.md（按需读取） ## 禁止事项 - 不直接修改 **/migration/** 下的 SQL 文件 - Service 层不跨模块直接调用其他 Service - 不在 controller 层写业务逻辑 ## 代码风格 - 遵循阿里 Java 开发手册 - 异常统一走 GlobalExceptionHandler
