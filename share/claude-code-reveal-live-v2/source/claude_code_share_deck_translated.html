02 技术栈选型 & 设计决策
层次技术选型为什么这样选？
运行时 Bun 比 Node.js 启动更快，原生 TypeScript 支持，feature() API 实现构建时 dead code 消除
语言 TypeScript strict 4600+ 文件规模下，类型安全不是选项而是必须
Terminal UI React 18 + Ink 权限对话框、进度条、多面板布局——终端 UI 足够复杂，React 组件模型合理
API 客户端 @anthropic-ai/sdk 官方 SDK，支持流式输出
工具扩展 MCP SDK 标准化 LLM 与外部工具通信协议
Feature Flag（功能开关） GrowthBook 服务端 A/B 测试 + 安全 kill switch，未发布功能门控
Java 类比：Bun ≈ GraalVM Native Image（快启动）；React/Ink ≈ JavaFX 但跑在终端；GrowthBook ≈ LaunchDarkly；TypeScript strict ≈ Java + Lombok + Bean Validation 一起用

04 核心：Agentic Loop（智能体主循环） 主循环

query.ts 函数签名

// 1729行的心脏 —— Async Generator（异步生成器）
export async function* query(
  params: QueryParams,
): AsyncGenerator<
  | StreamEvent
  | RequestStartEvent
  | Message
  | TombstoneMessage
  | ToolUseSummaryMessage,
  Terminal  // return: 终止原因
>
为什么 Async Generator（异步生成器）？ 把流式事件、正常终止、错误传播统一在单一函数里。Java 类比：Flux<T>（Reactor），但用语言原生语法，无需框架。

不可变参数 vs 可变状态

// 不可变参数 (整个循环不变)
type QueryParams = {
  messages: Message[]
  systemPrompt: SystemPrompt
  canUseTool: CanUseToolFn
  maxTurns?: number
}

// 可变状态 (每次迭代更新)
type State = {
  messages: Message[]
  turnCount: number
  transition: Continue | undefined
  // ...
}

// Continue Site 模式 — 原子替换
state = { ...state,
  messages: newMessages,
  turnCount: nextTurnCount,
  transition: { reason: 'next_turn' }
}

每个 Turn 的 6 阶段流水线

1
Pre-Request Compaction cheapest-first
按成本从低到高依次尝试 5 级压缩策略（详见 Slide 06）
2
API Call & Streaming（流式处理）
Streaming（流式处理）ToolExecutor：Claude 生成响应的同时并行执行只读工具，isConcurrencySafe 标志控制安全边界
3
Tool Authorization Pipeline（工具授权流水线）
PreFilter → PreToolUse Hook → Rule Evaluation → Permission Handler，每个工具独立鉴权
4
Tool Execution & Result Streaming（流式处理）
并发安全工具并行执行，写操作串行，结果流式回传
5
Recovery Mechanisms
token 超限自动重试，fallback model 切换，成本感知错误恢复
6
Stop Condition Check
maxTurns、用户中断、模型返回 end_turn、安全拒绝……任一满足则终止循环
05 工具系统：50+ 工具的设计哲学
核心原则：每个工具自带 input schema、permission level、execution logic，没有工具间共享的可变状态。BashTool 和 FileReadTool 在同一注册表中，但风险画像完全不同，架构分别对待。

并发安全控制核心逻辑

// Streaming（流式处理）ToolExecutor.ts
private canExecuteTool(isConcurrencySafe: boolean): boolean {
  const executing = this.tools.filter(t => t.status === 'executing')
  return (
    executing.length === 0 ||
    (isConcurrencySafe && executing.every(t => t.isConcurrencySafe))
  )
}
Java 类比：ReadWriteLock — 多读一写。只读工具拿 readLock 并发执行，写操作拿 writeLock 独占。但这里的"锁"是运行时调度逻辑，不是 JVM 锁。
06 Context 压缩管线：LLM 应用的核心挑战
问题本质：LLM 的 Context Window（上下文窗口） 是有限资源，长对话/大型任务会溢出。所有优化都围绕这个瓶颈展开。对话历史丢失后，客户端还需自行计算剩余 budget 并报给服务端——典型的分布式状态同步难题。

5 级压缩策略（cheapest-first 执行顺序）

Layer 1
Tool Result Budget（工具结果预算）
截断过大的工具结果（如文件读取返回数千行），保留关键内容
Layer 2
Snip Compact HISTORY_SNIP flag
直接丢弃最旧的消息块——最激进但成本最低
Layer 3
Microcompact
有 Prompt Cache（提示缓存） 感知地选择性清除单个工具结果，避免破坏缓存边界
Layer 4
Context Collapse（上下文折叠） CONTEXT_COLLAPSE flag
分阶段折叠消息块，生成压缩视图但不修改原始数据
Layer 5
Auto-Compact（自动摘要压缩）（LLM 摘要）
阈值触发，调用 LLM 对整段对话做摘要——最贵但最有效，Collapse 通过后跳过
关键设计：Context Collapse（上下文折叠） 必须在 Auto-Compact（自动摘要压缩） 之前执行——若 Collapse 已降至阈值以下，Auto-Compact（自动摘要压缩）（一次 API 调用费用）就不触发，保留细粒度上下文同时省钱。

Java GC 类比：Minor GC (L1-L2) → CMS (L3-L4) → Full GC (L5)，优先走成本最低的路径。
07 权限系统：Defense in Depth（纵深防御）

工具调用授权管线（4级）

1
Pre-filtering
按工具名和基础规则快速过滤，不允许的工具根本不进入后续流程
2
PreToolUse Hook
用户自定义脚本拦截点，可修改参数、记录日志、拒绝执行
3
Rule Evaluation
匹配 allowList/denyList 规则、路径边界检查、模式匹配
4
Permission Handler（人工确认）
不确定操作弹出 Terminal 对话框请求人工授权，auto-mode 用 ML 分类器决定是否可自动通过

3 种工作模式

Default Mode（默认模式）
每次不确定操作都询问用户，最高控制权，适合敏感操作
Auto Mode（自动模式）
ML 分类器评估风险，低风险自动执行，高风险仍需确认
Headless / CI Mode（无人值守/持续集成模式）
无人值守，依赖预设规则集，适合 CI/CD 流水线
Java 类比：Spring Security 的 SecurityFilterChain + 自定义 AccessDecisionVoter，但鉴权对象是"AI 将要执行的动作"而非"HTTP 请求"。Anthropic 的核心信念：Agent 的自主性正是其价值所在，但人类应始终保留对目标追求方式的控制权。
08 多 Agent 架构：隔离与委托

子 Agent 隔离模型

  • 独立 file cache（从父 Agent 克隆）
  • 独立 AbortController（异步）或共享（同步）
  • 独立 transcript 记录（JSONL sidechain）
  • 过滤后的工具池（按 Agent 定义限制）
  • 以纯文本形式向父 Agent 返回结果
  • 权限不继承——resume 时不恢复子 Agent 权限
Java 类比：Fork/Join 框架，但 worker 是独立的 LLM 实例，通信通道是自然语言而非方法调用返回值。

09 扩展架构：MCP + 四种机制
扩展机制类比（Java/Spring）作用层次典型用途
MCP Server SPI + gRPC/JSON-RPC 进程级隔离 数据库查询、Git 操作、自定义工具注册
Hooks（钩子机制） AOP 切面 / Filter 工具生命周期 审计日志、参数验证、安全扫描
Skills（技能包） @Configuration 模块 指令/知识 项目规范注入、领域知识复用
Plugins（插件） Maven Plugin 能力包 集成第三方服务（Jira、Slack 等）

为什么设计 4 种机制而不是 1 种？它们解决不同层面的扩展需求：MCP 扩展能力边界，Hooks（钩子机制） 扩展控制边界，Skills（技能包） 扩展知识边界，Plugins（插件） 扩展集成边界。强行统一会导致每种场景都被迫用不合适的抽象。

三个贯穿全局的架构选择

1. 分级分层（Graduated Layering）：每个关切（安全、权限、压缩、扩展）都有独立的分级机制，而非单一的单体实现。失败模式被隔离，不同级别可独立演进。
2. Append-Only（仅追加） 设计偏好可审计性：Session transcript 是 JSONL append-only 日志，支持 fork、resume、审计——查询能力换可观测性。类比 Event Sourcing。
3. 确定性 Harness（控制围栏） 内使用模型判断：架构不构建显式规划器，而是投资稳定的基础设施（context 管理、工具路由、错误恢复），让 LLM 在确定性 harness 内自主决策。
数据佐证：Anthropic 内部调研 132 名工程师，约 27% 的 Claude Code 辅助任务是"没有这个工具就不会尝试的工作"。架构实现的是质的工作流创新，而非仅仅加速现有流程。

6 个开放研究方向（学术论文总结）

  • 静默失败与可观测性缺口——Agent 失败时往往无声，需要更好的追踪和评估框架
  • 记忆作为一等公民——跨 session 的长期记忆目前依赖 CLAUDE.md，需要更系统的解决方案
  • Harness（控制围栏） 边界演化——在哪里、何时、做什么、与谁协作，边界如何随信任度动态调整
  • 架构解耦——推理、执行、会话三层进一步分离，允许独立扩展和替换
  • 治理与监督——随 Agent 自主性增强，如何在规模上维持人类监督
  • 主动架构——从响应式（等待输入）到主动式（主动感知、规划、行动）的转变
12 Java 开发者参考卡：概念映射
Claude Code 概念Java / Spring 生态类比本质差异
Async Generator（异步生成器） Loop Project Reactor Flux<T> 驱动源是 LLM 输出，而非数据流
Context Compaction（上下文压缩） 分级 GC (Minor/Major/Full) "内存"是 token，溢出代价是 API 费用
Permission Pipeline SecurityFilterChain + AccessDecisionVoter 鉴权对象是"AI 即将执行的动作"
isConcurrencySafe ReadWriteLock 调度逻辑在运行时，不是 JVM 锁
MCP Server SPI + JSON-RPC over gRPC 跨进程，通信通道是 stdio 或 HTTP SSE
Hooks（钩子机制） @Aspect + @Around 切入的是 LLM 工具调用生命周期
CLAUDE.md / Skills（技能包） @Configuration + application.yml 配置对象是 LLM 的行为约束，非 Bean
Session Transcript (JSONL) Event Sourcing / Kafka 日志 Append-Only（仅追加），支持 fork/resume/审计
SubAgent ForkJoinTask Worker 是独立 LLM 实例，通信是自然语言
Continue Site Pattern React setState / Immutable 对象 状态原子替换，防止中间状态污染
核心认知迁移：传统后端开发，你控制完整的执行流程。Agentic 开发，你构建"围栏"（harness）——确定性的安全边界、工具定义、权限规则，然后让 LLM 在围栏内自主决策。这是思维模式上最大的转变。
