# 交易系统工程能力学习计划

> 创建日期：2026-06-07
> 适用场景：以 MoreTickets 这类票务交易平台为样本，训练业务建模、系统链路和 AI 协作三类能力。
> 学习节奏：8 周，每周 3-5 次，每次 45-90 分钟；忙时只完成当周最低产出。

## 学习目标

这份计划不追求背概念，而是训练你能把复杂交易问题拆成清楚、可验证、可回滚的工程任务。

核心能力分三层：

1. 业务建模能力：能讲清订单、库存、支付、售后的状态机、幂等、补偿和并发一致性。
2. 系统链路能力：能分析 DB 事务、MQ、ES、缓存、外部供应商之间的数据一致性和失败恢复。
3. AI 协作能力：能把模糊问题描述成边界清晰、影响面明确、验收标准可检查、回滚方案可执行的任务。

## 使用方式

- 代码样本以 `~/code/moretickets` 为主，不直接改业务代码，先做阅读、画图、复盘和小范围实验。
- 每周输出一个 Markdown 笔记，建议放在本目录下，例如 `week-01-domain-map.md`。
- 每次学习按固定闭环推进：`找入口 -> 画状态/链路 -> 标风险 -> 写验证点 -> 写 AI 任务描述`。
- 任何不确定的接口、字段、枚举、状态含义，必须回到代码搜索确认，不靠记忆补全。

## 三类能力训练法

### 1. 业务建模能力

每看一个业务流程，都按这 8 个问题拆：

- 主实体是谁：用户订单、商家订单、履约单、支付单、售后单、退款单还是库存。
- 状态字段在哪里：枚举、数据库字段、DTO 字段是否一一对应。
- 状态由谁改变：Controller、App Service、Domain Entity、Consumer、Schedule、外部回调。
- 状态迁移条件是什么：权限、金额、库存、履约方式、售后原因、时间窗口。
- 幂等键是什么：订单号、支付流水号、退款单号、MQ message key、业务唯一索引。
- 并发保护是什么：数据库条件更新、唯一索引、乐观锁、分布式锁、状态前置条件。
- 补偿动作是什么：释放库存、关单、重推 MQ、重建 ES、退款、人工异常单。
- 哪些状态不可逆：支付成功、出票、履约完成、退款完成、结算完成等。

### 2. 系统链路能力

每条链路都画成这类结构：

```text
入口请求 / 外部回调
-> 参数校验和权限校验
-> App Service 编排
-> Domain 决策
-> Gateway 持久化
-> DB 事务提交
-> MQ 发送
-> Consumer 幂等处理
-> ES / Cache / Notification / Supplier
-> 查询端可见
```

重点追问：

- DB 是强一致源，ES 和 Cache 是否只是查询加速。
- MQ 是否在事务提交后发送，Consumer 是否能重复消费。
- ES 延迟时，前端看到旧数据是否可接受。
- 缓存更新是删除、写入还是 TTL 等待。
- 外部供应商返回失败、超时、重复回调时，系统以哪个本地状态为准。
- 定时任务和人工补偿是否会和实时链路并发。

### 3. AI 协作能力

## 重点实战题

按难度从低到高做，不要求一次做完：

1. 选一个订单查询接口，画出 `Controller -> QueryService -> Mapper/ES -> VO` 链路，并判断 ES 延迟的用户影响。
2. 选一个订单状态变更 Consumer，确认 topic/tag、幂等条件、失败重试策略和副作用。
3. 选一个库存锁定/释放流程，找出防重复释放和并发扣减的保护点。
4. 选一个支付回调流程，梳理签名校验、流水幂等、订单状态变更和后续 MQ。
5. 选一个售后退款流程，梳理售后单、退款单、支付事务和原订单的状态关系。
6. 选一个 ES Writer，确认 DB 字段到 Document 字段的映射，找出查询条件与构建字段是否一致。
7. 写一份“新增订单状态”的变更任务，明确兼容性、老数据、MQ、ES、前端展示和回滚方案。
8. 写一份“修复重复支付回调导致重复 MQ”的 Bug 任务，包含复现、根因、修复点和回归测试。

## 每周验收标准

一周学习合格的标准不是读了多少文件，而是能交付下面这些内容：

- 能用自己的话讲清一个状态机或系统链路。
- 能指出至少 3 个生产风险点。
- 能找到代码中的真实入口和真实保护点。
- 能写出至少 3 个验证场景：成功、失败、重复/并发。
- 能给 AI 写出一段可执行、可检查、可回滚的任务描述。

## 复盘问题

每周最后一次学习，用下面的问题复盘：

- 我本周画出的状态或链路，哪一段仍然只是猜测？
- 哪个状态迁移最容易被绕过？
- 哪个 MQ Consumer 最需要幂等保护？
- 哪个 ES/Cache 延迟会影响用户判断？
- 如果线上出问题，第一条止血动作是什么？
- 如果让 AI 改这个流程，我必须提前写清楚哪些边界？
