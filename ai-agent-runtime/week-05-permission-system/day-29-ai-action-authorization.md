## 4. 最小心智模型

可以先记住这句话：

> 模型提出动作后，谁决定能不能执行？ 这个问题的答案，最终都要落到“如何让 Agent 更可控、更准确、更可验证”。

从 Runtime 视角看，今天主题和下面链路有关：

```text
User Goal
-> Context / State
-> Model Decision
-> Runtime Control
-> Tool / Memory / Permission / Trace
-> Observation
-> Next Step
```

不要只问“模型会不会”，要问：

- Runtime 给模型看了什么？
- 模型输出如何被解析和校验？
- 工具或状态是否真的发生变化？
- 失败时有没有记录和恢复？
- 最终结论有没有证据？

## 5. 核心概念拆解

### 5.1 Authorization（授权）

判断动作是否允许执行。

它解决的问题：让 `Authorization（授权）` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `Authorization（授权）`。

忽略后果：如果 `Authorization（授权）` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

### 5.2 Risk Classification（风险分类）

## 6. 工程含义

今天主题的工程含义可以分成 5 层：

1. **边界**：明确模型、Runtime、工具、状态、用户各自负责什么。
2. **结构**：用接口、Schema、状态机、表结构或日志结构把能力固定下来。
3. **安全**：对高风险动作设置权限、审批、沙箱或只读限制。
4. **可恢复**：失败后能重试、降级、停止或交给用户处理。
5. **可验证**：最终结论必须能从工具结果、日志、状态或测试中找到证据。

## 7. Java / 后端类比

## 11. 常见坑

- 只会解释概念，但说不出它在 Runtime 里的位置。
- 只相信模型输出，没有结构化校验。
- 没有考虑失败、超时、权限和审计。
- 把所有信息都塞进上下文，导致模型被噪声干扰。
- 没有最终验证，却在回答里声称任务完成。

## 13. 补充深度学习内容

### 13.1 动作授权不是“问模型能不能做”

`Authorization（授权）` 是 Runtime 的安全决策，不是模型的自我约束。

模型可以提出：

```text
我要执行 delete_file(path="/tmp/a.txt")
```

但 Runtime 必须判断：

- 这个工具是否存在？
- path 是否在允许范围内？
- 动作是只读还是写操作？
- 用户是否有权限？
- 是否需要人工确认？
- 是否有可回滚方案？

### 13.2 风险等级表

| 等级 | 动作 | 策略 |
|------|------|------|
| LOW | 只读搜索、读取公开文件 | 自动执行并记录 |
| MEDIUM | 写临时文件、运行测试 | 可自动执行，但限制范围 |
| HIGH | 删除文件、修改配置、发消息 | 必须人工确认 |
| CRITICAL | 生产 DB、支付退款、发布上线 | 默认拒绝或走正式审批 |

### 13.3 最小设计

```java
class AuthorizationDecision {
    boolean allowed;
    RiskLevel riskLevel;
    boolean requiresApproval;
    String reason;
    List<String> constraints;
}
```

今日要记住：模型只有建议权，Runtime 才有授权权。
