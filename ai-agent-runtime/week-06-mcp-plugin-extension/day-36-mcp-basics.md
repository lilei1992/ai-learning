## 4. 最小心智模型

可以先记住这句话：

> MCP 解决 Agent 工具扩展中的什么问题？ 这个问题的答案，最终都要落到“如何让 Agent 更可控、更准确、更可验证”。

从 Runtime 视角看，今天主题和下面链路有关：

```text
User Goal
-> Context / State
-> Model Decision
-> Runtime Control
-> Tool / Memory / Permission / Trace
-> Observation
-> Next Step
```

不要只问“模型会不会”，要问：

- Runtime 给模型看了什么？
- 模型输出如何被解析和校验？
- 工具或状态是否真的发生变化？
- 失败时有没有记录和恢复？
- 最终结论有没有证据？

## 5. 核心概念拆解

### 5.1 MCP（Model Context Protocol）

模型应用连接外部工具和资源的协议。

它解决的问题：让 `MCP（Model Context Protocol）` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `MCP（Model Context Protocol）`。

忽略后果：如果 `MCP（Model Context Protocol）` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

### 5.2 Host（宿主）

运行 Agent 的应用。

它解决的问题：让 `Host（宿主）` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `Host（宿主）`。

忽略后果：如果 `Host（宿主）` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

### 5.3 Server（服务端）

提供工具或资源的外部进程。

它解决的问题：让 `Server（服务端）` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `Server（服务端）`。

忽略后果：如果 `Server（服务端）` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

### 5.4 Capability Discovery（能力发现）

让 Agent 知道有哪些工具可用。

它解决的问题：让 `Capability Discovery（能力发现）` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `Capability Discovery（能力发现）`。

忽略后果：如果 `Capability Discovery（能力发现）` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

## 6. 工程含义

今天主题的工程含义可以分成 5 层：

1. **边界**：明确模型、Runtime、工具、状态、用户各自负责什么。
2. **结构**：用接口、Schema、状态机、表结构或日志结构把能力固定下来。
3. **安全**：对高风险动作设置权限、审批、沙箱或只读限制。
4. **可恢复**：失败后能重试、降级、停止或交给用户处理。
5. **可验证**：最终结论必须能从工具结果、日志、状态或测试中找到证据。

## 8. 设计清单

学习今天主题时，至少检查这些设计点：

- 是否有清晰的输入和输出。
- 是否有结构化数据，而不是只靠自然语言。
- 是否能被记录到 Transcript / Trace。
- 是否能区分成功、失败、拒绝、超时和部分成功。
- 是否需要权限控制。
- 是否需要幂等或重试。
- 是否会污染上下文或 Memory。
- 是否能被测试和回放。

## 11. 常见坑

- 只会解释概念，但说不出它在 Runtime 里的位置。
- 只相信模型输出，没有结构化校验。
- 没有考虑失败、超时、权限和审计。
- 把所有信息都塞进上下文，导致模型被噪声干扰。
- 没有最终验证，却在回答里声称任务完成。

## 12. 今日总结

今天真正要记住的是：

> Agent 工程化不是让模型“更自由”，而是让模型的推理能力被 Runtime 安全、结构化、可追踪地使用。

## 13. 补充深度学习内容

### 13.1 MCP 是 Agent 的 USB-C

`MCP（Model Context Protocol，模型上下文协议）` 的价值是让 Runtime 用统一方式接入外部能力。

它解决：

- 工具发现。
- 参数 Schema 描述。
- 资源读取。
- Prompt 模板暴露。
- 跨进程调用。
- 错误结构化。

### 13.2 MCP 不等于安全

MCP 提供协议，不自动提供权限、安全和审计。Runtime 仍然要做授权、沙箱和 Trace。
