> 为什么插件和工具最好隔离运行？ 这个问题的答案，最终都要落到“如何让 Agent 更可控、更准确、更可验证”。

从 Runtime 视角看，今天主题和下面链路有关：

```text
User Goal
-> Context / State
-> Model Decision
-> Runtime Control
-> Tool / Memory / Permission / Trace
-> Observation

```

不要只问“模型会不会”，要问：

- Runtime 给模型看了什么？

- 最终结论有没有证据？

## 5. 核心概念拆解

### 5.1 Process Isolation（进程隔离）

工具在独立进程中运行，降低崩溃和越权影响。

它解决的问题：让 `Process Isolation（进程隔离）` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `Process Isolation（进程隔离）`。

忽略后果：如果 `Process Isolation（进程隔离）` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

### 5.2 Crash Boundary（崩溃边界）

插件挂了不拖垮主应用。

它解决的问题：让 `Crash Boundary（崩溃边界）` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `Crash Boundary（崩溃边界）`。

忽略后果：如果 `Crash Boundary（崩溃边界）` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

### 5.3 Permission Boundary（权限边界）

不同插件拥有不同权限。

它解决的问题：让 `Permission Boundary（权限边界）` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `Permission Boundary（权限边界）`。

忽略后果：如果 `Permission Boundary（权限边界）` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

### 5.4 Resource Limit（资源限制）

控制 CPU、内存、时间。

它解决的问题：让 `Resource Limit（资源限制）` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `Resource Limit（资源限制）`。

忽略后果：如果 `Resource Limit（资源限制）` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

## 6. 工程含义

今天主题的工程含义可以分成 5 层：

1. **边界**：明确模型、Runtime、工具、状态、用户各自负责什么。
2. **结构**：用接口、Schema、状态机、表结构或日志结构把能力固定下来。
3. **安全**：对高风险动作设置权限、审批、沙箱或只读限制。
4. **可恢复**：失败后能重试、降级、停止或交给用户处理。
5. **可验证**：最终结论必须能从工具结果、日志、状态或测试中找到证据。

## 11. 常见坑

- 只会解释概念，但说不出它在 Runtime 里的位置。
- 只相信模型输出，没有结构化校验。
- 没有考虑失败、超时、权限和审计。
- 把所有信息都塞进上下文，导致模型被噪声干扰。
- 没有最终验证，却在回答里声称任务完成。

## 12. 今日总结

今天真正要记住的是：

> Agent 工程化不是让模型“更自由”，而是让模型的推理能力被 Runtime 安全、结构化、可追踪地使用。

## 13. 补充深度学习内容

### 13.1 插件不应运行在 Runtime 主进程

外部 MCP Server 可能崩溃、卡死、输出脏数据或读取不该读的资源。

进程隔离带来：

- 可单独启动和停止。
- 可设置超时。
- 可限制环境变量。
- 可隔离崩溃影响。
- 可记录 stdout/stderr。

### 13.2 生命周期管理

Runtime 要管理插件进程的 start、health check、restart、timeout、shutdown。
