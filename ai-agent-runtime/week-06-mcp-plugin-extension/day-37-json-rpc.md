## 4. 最小心智模型

可以先记住这句话：

> MCP 工具调用如何通过 JSON-RPC 表达？ 这个问题的答案，最终都要落到“如何让 Agent 更可控、更准确、更可验证”。

从 Runtime 视角看，今天主题和下面链路有关：

```text
User Goal
-> Context / State
-> Model Decision
-> Runtime Control
-> Tool / Memory / Permission / Trace
-> Observation
-> Next Step
```

不要只问“模型会不会”，要问：

- Runtime 给模型看了什么？
- 模型输出如何被解析和校验？
- 工具或状态是否真的发生变化？
- 失败时有没有记录和恢复？
- 最终结论有没有证据？

## 5. 核心概念拆解

### 5.1 JSON-RPC

用 JSON 表达远程方法调用。

它解决的问题：让 `JSON-RPC` 从抽象术语变成 Runtime 可以判断、记录和执行的工程边界。

工程落点：通常需要在状态字段、策略规则、接口 Schema、执行流程或审计记录中明确表达 `JSON-RPC`。

忽略后果：如果 `JSON-RPC` 只停留在 prompt 里，Agent 容易出现误执行、假成功、不可恢复或不可追踪的问题。

## 6. 工程含义

今天主题的工程含义可以分成 5 层：

1. **边界**：明确模型、Runtime、工具、状态、用户各自负责什么。
2. **结构**：用接口、Schema、状态机、表结构或日志结构把能力固定下来。
3. **安全**：对高风险动作设置权限、审批、沙箱或只读限制。
4. **可恢复**：失败后能重试、降级、停止或交给用户处理。
5. **可验证**：最终结论必须能从工具结果、日志、状态或测试中找到证据。

## 11. 常见坑

- 只会解释概念，但说不出它在 Runtime 里的位置。
- 只相信模型输出，没有结构化校验。

## 13. 补充深度学习内容

### 13.1 JSON-RPC 是跨进程消息格式

MCP 常基于 `JSON-RPC` 表达请求和响应：

```json
{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}
```

它的价值：

- 请求有 id，响应可匹配。
- 错误有标准结构。
- 通知不需要响应。
- 适合 stdio、HTTP 等传输。

### 13.2 Runtime 要处理协议异常

包括超时、无响应、非法 JSON、重复 id、未知 method、错误码映射。
