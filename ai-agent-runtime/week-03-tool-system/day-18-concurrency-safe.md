## 5. 核心概念拆解

### 5.1 Read / Write Split（读写分离）

只读工具可并发，写工具通常要串行。

进一步理解这个概念时，建议追问三件事：

- 它解决的问题：避免 Agent 在缺少结构、缺少证据或缺少边界的情况下行动。
- 工程落点：它通常会落到接口、Schema、状态字段、策略规则、日志字段或执行流程中。
- 忽略后果：模型可能继续基于错误前提行动，造成假成功、越权、上下文污染或不可追踪失败。

## 6. 工程含义

今天主题的工程含义可以分成 5 层：

1. **边界**：明确模型、Runtime、工具、状态、用户各自负责什么。
2. **结构**：用接口、Schema、状态机、表结构或日志结构把能力固定下来。
3. **安全**：对高风险动作设置权限、审批、沙箱或只读限制。
4. **可恢复**：失败后能重试、降级、停止或交给用户处理。
5. **可验证**：最终结论必须能从工具结果、日志、状态或测试中找到证据。

## 13. 补充深度学习内容

### 13.1 Agent 工具为什么会有并发问题

如果 Runtime 支持并行工具调用，可能出现：

- 两个工具同时写同一个文件。
- 一个工具读文件时另一个工具正在修改。
- 多个测试命令同时占用构建资源。
- 用户取消任务时工具仍在运行。
- 多个 Agent 同时操作同一个 workspace。

并发不是“跑快一点”这么简单，它会直接影响正确性。

### 13.2 工具并发分级

| 工具类型 | 是否适合并发 | 原因 |
|----------|--------------|------|
| `read_file` | 适合 | 只读，低风险 |
| `search_code` | 适合 | 只读，低风险 |
| `query_log` | 适合 | 只读，可限流 |
| `run_test` | 视情况 | 可能占用构建资源 |
| `write_file` | 不适合随意并发 | 有写冲突 |
| `delete_file` | 不适合 | 高风险 |
| `git commit` | 不适合 | 依赖工作区一致性 |
| `update_database` | 不适合 | 业务副作用大 |

### 13.3 Resource Lock 设计

```java
public class ToolResourceLock {
    private String resourceType; // file, workspace, database, git
    private String resourceId;
    private LockMode mode; // READ, WRITE
    private String ownerTaskId;
}
```

锁粒度建议：

- 文件读取：`file:path` 读锁。
- 文件写入：`file:path` 写锁。
- Git 操作：`workspace` 写锁。
- 数据库写操作：默认禁止或业务主键锁。

### 13.4 并发安全策略

```text
只读工具：
  可并发，但要限流和超时。

写文件工具：
  同一文件串行。
  同一 workspace 写操作谨慎并发。

Git 工具：
  串行。

高风险外部操作：
  默认不并发，且需要人工确认。
```

### 13.5 取消和部分执行

用户点击取消时，Runtime 需要：

1. 标记任务为 `CANCELLING`。
2. 停止调度新工具。
3. 尝试取消正在运行的工具。
4. 收集已完成工具结果。
5. 标记是否存在部分写入。
6. 最终进入 `CANCELLED`。

不能简单把任务状态改成 cancelled，却让后台工具继续写文件。

### 13.6 今日实践输出方向

设计一个并发策略表：

```text
Tool: read_file
Lock: file read lock
Concurrent: yes

Tool: write_file
Lock: file write lock
Concurrent: same file no

Tool: git_commit
Lock: workspace write lock
Concurrent: no
```

这张表后续可以变成 ToolDefinition 的一部分。
