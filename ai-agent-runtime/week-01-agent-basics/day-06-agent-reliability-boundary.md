## 4. 最小心智模型

Agent 的可靠性不是：

```text
模型足够聪明 -> 任务自然完成
```

而是：

```text
模型提出下一步
-> Runtime 校验动作
-> 工具安全执行
-> 结果可观察
-> 失败可恢复
-> 完成可验证
```

一句话：

> 模型负责提出可能的行动，Runtime 负责让行动安全、稳定、可审计。

## 5. Agent 的主要不可靠来源

### 5.1 模型输出不稳定

同一个目标，模型可能在不同时间给出不同路径。

风险：

- 工具选择不一致。
- 参数生成错误。
- 忘记前面失败过的尝试。
- 过早声称完成。

工程应对：

- 明确工具 schema。
- 设置最大轮数。
- 保留关键轨迹。
- 对最终结果做验证。

### 5.2 工具执行失败

工具失败可能来自：

- 文件不存在。
- 命令超时。
- 网络异常。
- 权限不足。
- 参数不合法。
- 外部服务返回错误。

工程应对：

- 工具结果必须带成功/失败状态。
- 错误信息要可诊断。
- 可重试错误和不可重试错误要区分。
- 不允许把失败包装成成功。

### 5.3 上下文不完整或被污染

Agent 的判断依赖上下文。上下文不完整会导致误判，上下文污染会导致偏航。

风险：

- 使用过期文件内容。
- 引用未验证假设。
- 被长日志淹没关键事实。
- 忘记用户最新要求。

工程应对：

- 控制工具结果长度。
- 保留关键约束。
- 对事实来源做标记。
- 必要时重新读取源文件。

### 5.4 写操作有副作用

读错信息通常还能纠正，写错文件、发错请求、删错数据会产生真实损害。

写操作风险：

- 重复执行造成重复副作用。
- 部分成功导致状态不一致。
- 未授权修改敏感文件。
- 修改后没有验证。

工程应对：

- 写前权限检查。
- 写后 diff 检查。
- 幂等设计。
- 可回滚或可人工审查。
- 关键动作要求用户确认。

## 6. 关键可靠性手段

### 6.1 Guardrail（护栏）

`Guardrail` 是限制 Agent 行为的规则和机制。

例子：

- 禁止访问仓库外敏感目录。
- 写操作前必须检查 diff。
- 删除文件需要人工确认。
- 禁止输出未验证的“测试已通过”。
- 禁止记录 token、cookie、password。

护栏不是为了阻止 Agent 工作，而是为了让它在安全边界内工作。

### 6.2 Fallback（降级）

`Fallback` 是失败后的保守策略。

例子：

- 搜索失败时改为读取目录索引。
- 测试命令不可用时报告未验证原因。
- 模型连续生成非法工具参数时停止并询问用户。
- 外部服务超时时返回可重试失败。

好的降级应该：

- 明确说明原路径为什么失败。
- 不伪装成成功。
- 给出下一步可行动建议。

### 6.3 Idempotency（幂等性）

`Idempotency` 表示同一动作重复执行不会造成额外副作用。

Agent 写操作必须考虑幂等，因为：

- 模型可能重复请求同一工具。
- Runtime 可能因为超时重试。
- 会话恢复后可能重放动作。
- 用户可能让 Agent 再执行一次。

例子：

- 写文件前先比较内容，相同则不重复写。
- 创建资源时使用唯一业务键。
- MQ 消费使用去重表。
- 外部 API 使用 idempotency key。

### 6.4 Verification（验证）

Agent 不能只因为“已经执行过动作”就声称完成。

验证方式：

- 运行测试。
- 检查命令退出码。
- 查看 diff。
- 重新读取文件。
- 调用查询接口确认状态。
- 让用户确认无法自动验证的业务语义。

可靠 Agent 的关键习惯：

> 没有验证证据，就不能声称成功。

### 6.5 Audit（审计）

审计记录至少包括：

- 用户目标。
- 模型请求的动作。
- 工具名称和参数摘要。
- 权限决策。
- 执行结果。
- 错误和重试。
- 最终停止原因。

审计不是事后形式主义，它是调试、追责、评估和安全治理的基础。

## 7. 假成功：Agent 最危险的失败之一

假成功指：

> Agent 输出“已完成”“已修复”“测试通过”，但实际没有完成、没有修复或没有验证。

常见原因：

- 没有运行测试却说测试通过。
- 工具失败后只看到了部分输出。
- 把计划当成执行结果。
- 修改了文件但没有保存。
- 保存了文件但没有检查 diff。
- 忽略权限拒绝或命令退出码。

应对方式：

- 最终回答必须区分“已验证”和“未验证”。
- 工具结果中保留退出码和错误摘要。
- Runtime 或系统指令禁止无证据成功声明。
- 重要动作完成后必须有验证步骤。

## 8. Java / 后端类比

| Agent 可靠性概念 | Java / 后端类比 |
|------------------|------------------|
| Guardrail | 参数校验 / 权限校验 / 状态机约束 |
| Fallback | Feign fallback / 熔断降级 |
| Idempotency | MQ 消费幂等 / 支付幂等 |
| Verification | 单元测试 / 集成测试 / 回查数据库 |
| Audit | 操作日志 / 安全审计 / trace log |
| maxTurns | 最大重试次数 / 防死循环保护 |
| timeout | RPC 超时 / 任务超时 |

相似点：

- 都要承认外部调用会失败。
- 都要防止重复副作用。
- 都要记录关键动作。
- 都要区分成功、失败、处理中。

不同点：

- 后端代码路径通常固定，Agent 路径可能动态生成。
- Agent 的错误可能来自模型推断，不只是系统异常。
- Agent 更容易出现“语言上说成功，工程上没成功”的错位。

## 5. 核心概念拆解（标准化补充）

- Guardrail（护栏）：事前或事中限制危险行为。
- Idempotency（幂等性）：重复执行不会造成重复副作用。
- Fallback（兜底）：失败后切换到安全替代方案。
- Verified Completion（已验证完成）：完成结论有工具结果或状态证据支撑。

## 9. 今日实践任务（标准化补充）

为一个写文件工具设计可靠性清单：权限、幂等、备份、diff、验证、审计。

## 10. 自测问题与参考答案（标准化补充）

### Q1：假成功为什么危险？

用户以为任务完成，但真实状态未达成，后续决策会基于错误前提。

### Q2：写操作为什么必须考虑幂等？

Agent 可能重试或恢复，非幂等写操作会放大副作用。

## 11. 常见坑（标准化补充）

- 没有验证就说完成。
- 高风险动作无审批。
- 重试非幂等操作。
- 审计日志泄露敏感信息。

## 12. 今日总结（标准化补充）

可靠性来自 Runtime 的权限、状态、幂等、审计、验证和恢复，不来自模型自觉。

## 13. 补充深度学习内容

生产级 Agent 的难点不是让模型生成正确答案，而是让错误、超时、拒绝、部分成功和不确定性都能被系统化处理。
