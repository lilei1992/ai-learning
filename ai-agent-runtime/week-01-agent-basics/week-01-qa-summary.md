# Week 01：Agent 基础模型问题汇总

> 用途：第一周复习用。  
> 原则：按天分模块，答案保持精简，优先记住概念边界和工程判断。

## Day 01：AI Agent Runtime（智能体运行时）

### Q1：`LLM` 和 `Agent` 的区别是什么？

`LLM（Large Language Model，大语言模型）` 是负责理解、推理和生成文本的模型；`Agent（智能体）` 是围绕目标持续行动的系统。简单说：`LLM` 是“大脑”，`Agent` 是“会做事的系统”。

### Q2：`Runtime` 为什么不是普通 UI 层？

UI 负责展示和交互；`Runtime（运行时）` 负责上下文组装、模型调用、工具执行、权限控制、状态管理、错误恢复和执行记录。它是 Agent 的执行内核，不是页面组件。

### Q3：`Observation（观察结果）` 在 Agent Loop 中有什么作用？

Observation 是工具执行后的真实反馈，用来支撑下一轮决策。没有 Observation，Agent 就只能靠猜，无法基于现实结果迭代。

### Q4：为什么 Chat API 的主要产物是文本，而 Agent Runtime 的产物可能是状态变化？

`Chat API（聊天接口）` 主要返回回答、摘要、代码片段等文本；`Agent Runtime` 可能创建文件、修改代码、运行命令、提交 Git、更新任务状态。它的完成标准通常是外部状态真的改变。

### Q5：如果模型说“我要执行 `rm -rf .`”，Runtime 应该怎么处理？

识别为高风险破坏性动作，默认拦截；记录原因；必要时要求用户显式确认；建议更安全的替代方案。模型只能提出动作，不能等同于授权。

### Q6：为什么每次工具调用都应该进入 `Transcript（执行记录）`？

为了可追溯、可审计、可恢复、可调试。Transcript 让每一步从“模型说过”变成“系统可验证”。

### Q7：Agent Runtime 和传统工作流引擎有什么相似点和区别？

相似点：都做任务编排、状态管理、错误处理和审计。区别：Workflow 执行预设流程；Agent Runtime 允许模型根据目标和 Observation 动态决定下一步。

### Q8：工具返回结果很长时，Runtime 应该全部塞回上下文吗？

不应该。Runtime 应该过滤、摘要、脱敏、结构化，只保留与下一步决策相关的信息，避免上下文超限、噪声干扰、成本上升和敏感信息泄露。

### Q9：Agent 说“完成了”，如何验证真的完成？

看证据，不看声明。检查目标条件、Transcript、Observation、外部系统状态和副作用。完成 = 条件满足 + 有证据 + 状态已验证 + 无已知阻塞问题。

## Day 02：Chat API、Workflow、Agent

### Q1：Chat API、Workflow、Agent 的一句话区别是什么？

`Chat API` 负责生成回答；`Workflow（工作流）` 负责按预设步骤执行；`Agent` 负责根据目标和观察结果动态推进任务。

### Q2：为什么 Chat API 的主要产物通常是文本？

因为它的基本模式是 `messages -> model -> response`，默认不执行外部动作，也不改变文件、数据库或浏览器状态。

### Q3：Workflow 为什么比 Agent 更适合订单、支付、退款这类业务流程？

这些流程规则明确、风险高、需要状态机、幂等、补偿和审计。不能让模型自由决定核心状态变化。

### Q4：Agent 为什么更适合分析陌生代码库？

因为路径不确定，需要搜索、阅读、运行命令、根据发现继续深入。固定 Workflow 很难提前穷举所有步骤。

### Q5：Workflow 中能不能使用 LLM？

可以，但 LLM 应该作为某个节点使用，例如分类、摘要、生成说明；不能绕过 Workflow 的状态控制和权限校验。

### Q6：为什么生产系统常用 `Workflow + Agent`？

Agent 负责探索、理解和辅助执行；Workflow 负责确定性状态流转、审计和补偿。这样既有灵活性，也有可靠性。

### Q7：`Dynamic Planning（动态规划）` 和 `Deterministic Flow（确定性流程）` 的区别是什么？

Dynamic Planning 是模型根据当前上下文和观察结果动态决定下一步；Deterministic Flow 是代码规则预先决定流程路径。

## Day 03：ReAct（Reasoning and Acting，推理与行动）

### Q1：ReAct 的三个核心环节是什么？

`Reasoning（推理） -> Acting（行动） -> Observation（观察）`。工程上常表现为 `Thought -> Action -> Observation -> Next Thought`。

### Q2：为什么 Observation 必须进入下一轮上下文？

因为下一步决策必须基于真实反馈。没有 Observation，模型不知道工具是否成功、错误在哪里、任务是否完成。

### Q3：Runtime 在 ReAct 中至少负责哪些事情？

上下文组装、模型调用、动作解析、权限控制、工具执行、Observation 整理、状态存储、停止条件和 Trace / Log。

### Q4：为什么 Agent 不应该一看到错误就直接修改代码？

错误现象不等于根因。可能是实现错、测试过期、业务规则变化、Mock 错或环境问题。应该先复现、读取代码、确认语义，再决定是否修改。

### Q5：ReAct 和固定 Workflow 的核心区别是什么？

Workflow 按预设路线执行；ReAct 边观察边决定路线。Workflow 更稳定，ReAct 更适合探索性任务。

### Q6：工具执行失败时，Observation 应该如何表达？

应包含状态、动作、关键错误、证据和下一步约束。例如 `status=failed`、命令、失败测试名、关键日志、是否允许重试。

### Q7：模型连续 3 次搜索同一个关键词，Runtime 应该怎么处理？

识别为循环或低效行为，阻止重复搜索，返回 Observation，要求换关键词、读取已有结果、总结证据或询问用户。

### Q8：ReAct 为什么比单次 Chat API 更适合代码修复？

代码修复需要运行测试、读取日志、定位文件、修改代码、再次验证。ReAct 能把每次工具反馈用于下一轮行动。

### Q9：ReAct 轨迹对审计和回归测试有什么价值？

轨迹记录目标、动作、参数、结果和决策依据，可用于复盘、审计、行为回归测试和判断 Agent 是否跳过关键步骤。

### Q10：什么情况下 Agent 应该停止并询问用户？

业务语义不明确、操作高风险、权限不足、多轮无进展、目标冲突、外部系统不稳定或需要用户做价值判断时。

## Day 04：State Machine（状态机）与 Agent

### Q1：Agent 的 State 至少应该保存哪些信息？

目标、当前状态、轮次、历史动作、Observation、错误、权限结果、已修改资源、停止原因和最终结果。

### Q2：`maxTurns（最大轮数）` 为什么必要？

防止 Agent 无限循环、重复搜索、持续调用工具导致成本和风险失控。它是停止条件的一部分。

### Q3：什么是 `Terminal State（终止状态）`？

任务不会再继续转移的状态，例如 `DONE（完成）`、`FAILED（失败）`、`CANCELLED（取消）`、`BLOCKED（阻塞）`。

### Q4：为什么工具失败不能被吞掉？

吞掉失败会制造假成功，导致模型基于错误前提继续行动。失败必须进入 State、Observation 和 Transcript。

### Q5：状态机如何减少 Agent 的不可控行为？

通过显式状态、合法转移、不变量和停止条件限制 Agent 行为，避免乱跳步骤、重复执行和未授权动作。

### Q6：写文件工具执行中用户点击取消，状态机应该如何转移？

应进入 `CANCELLING` 或 `CANCELLED`；如果写操作已部分完成，需要记录 partial result，并提示是否需要回滚或人工检查。

### Q7：模型输出非法工具参数，应该进入什么状态？

进入 `ACTION_INVALID`、`FAILED` 或可恢复的 `WAITING_MODEL_RETRY`，具体取决于策略。关键是不能直接执行。

### Q8：为什么 `DONE` 和 `FAILED` 都是终止状态但含义不同？

`DONE` 表示目标已满足并验证；`FAILED` 表示任务无法完成或出现不可恢复错误。二者都停止，但结论完全不同。

### Q9：Agent 状态不变量和订单系统不变量有什么相似点？

都用于保护核心语义。例如订单不能从已退款直接变已支付；Agent 不能从未授权动作直接进入工具执行。

### Q10：没有 `stopReason（停止原因）` 会有什么问题？

无法判断任务是完成、失败、超时、取消、权限不足还是达到最大轮数，排查和审计都会困难。

## Day 05：Message、Memory、Transcript

### Q1：Message、Memory、Transcript 的一句话区别是什么？

`Message（消息）` 是当前对话内容；`Memory（记忆）` 是跨会话保留的稳定信息；`Transcript（执行记录）` 是工具调用和状态变化的事实账本。

### Q2：为什么 Memory 写入必须谨慎？

Memory 会长期影响后续决策。把错误、临时、未验证或敏感信息写入 Memory，会造成长期污染。

### Q3：Transcript 为什么不等于 Message？

Message 面向对话上下文；Transcript 面向审计和恢复。Transcript 可以很详细，但不一定全部进入模型上下文。

### Q4：工具结果默认应该进入哪里？

默认进入 Transcript；经过筛选、摘要后，关键 Observation 可以进入下一轮上下文。只有稳定偏好或长期事实才可能进入 Memory。

### Q5：哪些信息适合跨会话保留？

用户稳定偏好、长期项目背景、明确确认过的规则、常用路径和经过验证的事实。

### Q6：用户说“以后都用中文回答”，应该写入哪里？

应写入 Memory，也会作为当前 Message 和 Transcript 的一部分被记录。它是稳定用户偏好。

### Q7：一次测试失败日志应该写入 Memory 吗？

通常不应该。测试日志是临时执行结果，应写入 Transcript；只有提炼出的长期规则或已确认事实才考虑写入 Memory。

### Q8：Agent 误把未验证猜测写入 Memory 会有什么后果？

后续任务会被错误事实污染，模型可能反复基于错误前提推理，导致难以排查的长期偏差。

### Q9：Transcript 对问题排查有什么价值？

它能还原 Agent 做过什么、参数是什么、结果如何、哪里失败、是否越权，支持调试、恢复和审计。

### Q10：为什么不是所有 Transcript 都应该进入下一轮上下文？

因为 Transcript 可能很长、含噪声或敏感信息。下一轮上下文只需要相关、精简、脱敏后的 Observation。

## Day 06：Agent 可靠性边界

### Q1：为什么 Agent 可靠性不能只靠模型能力？

模型会误判、幻觉、重复行动或忽略失败。可靠性需要 Runtime 的权限、状态、校验、重试、审计和验证共同保证。

### Q2：写操作为什么必须考虑幂等？

因为 Agent 可能重试、恢复或重复调用工具。幂等能避免重复创建、重复扣款、重复删除、重复提交等副作用。

### Q3：`Guardrail（护栏）` 和 `Fallback（兜底）` 的区别是什么？

Guardrail 是事前或事中限制危险行为；Fallback 是失败后切换到更安全方案、降级处理或请求用户介入。

### Q4：假成功为什么危险？

假成功会让用户以为任务完成，但真实状态未达成；后续决策会基于错误前提，可能造成数据、代码或业务风险。

### Q5：工具结果里至少应该保留哪些诊断信息？

工具名、输入摘要、执行状态、关键输出、错误信息、耗时、目标资源、权限结果和可追溯引用。

### Q6：如果 Agent 没有运行测试，最终回答应该如何表达？

必须明确说“未运行测试”，不能声称已验证。可以说明原因，并给出建议执行的验证命令。

### Q7：工具超时时，Runtime 应该重试、降级还是停止？

取决于操作是否安全幂等、失败原因、重试次数、超时成本、业务风险和用户授权。非幂等高风险操作不能盲目重试。

### Q8：为什么删除文件这类动作需要更高权限控制？

删除是破坏性操作，可能不可恢复、影响范围大、路径易误判。必须确认目标、范围、权限和回滚方案。

### Q9：幂等性和重试机制是什么关系？

重试依赖幂等保证安全；没有幂等，重试可能放大副作用。先设计幂等，再设计自动重试。

### Q10：Agent 审计日志应该避免记录哪些敏感信息？

Token、密码、Cookie、私钥、Authorization Header、个人隐私、生产敏感数据和完整密钥配置。

## Day 07：Week 01 复盘

### Q1：为什么说 Runtime 是 Agent 系统的执行内核？

因为 Runtime 负责把模型输出变成受控行动，并管理上下文、工具、权限、状态、日志、错误恢复和验证。

### Q2：Workflow 和 Agent 各适合什么场景？

Workflow 适合规则明确、风险高、流程稳定的业务；Agent 适合路径不确定、需要探索和多步工具反馈的任务。

### Q3：ReAct 中 Observation 质量差会造成什么问题？

会导致模型抓错重点、重复行动、忽略失败、上下文污染，甚至错误声称完成。

### Q4：Agent 状态机至少需要哪些终止状态？

至少需要 `DONE`、`FAILED`、`CANCELLED`、`BLOCKED`、`TIMEOUT` 或 `MAX_TURNS_REACHED`。

### Q5：Memory 写入策略为什么要保守？

因为 Memory 是长期影响源，错误写入会持续污染后续任务。只写稳定、明确、经过确认的长期信息。

### Q6：Transcript 对审计和恢复有什么价值？

它记录每次工具调用、参数、结果、错误和权限判断，可用于证明、回放、恢复和定位问题。

### Q7：写操作为什么需要权限、幂等和验证？

权限防止越权，幂等防止重复副作用，验证确认真实状态符合目标。三者缺一不可。

### Q8：Agent 声称“任务完成”前应该提供什么证据？

应提供执行过的关键动作、工具结果、验证命令或检查结果、外部状态证据，以及未验证项说明。

## 本周最小复习清单

- `LLM` 生成能力强，但不是完整 Agent。
- `Runtime` 是 Agent 的执行控制层。
- `Workflow` 适合确定流程，`Agent` 适合开放任务。
- `ReAct` 的核心是推理、行动、观察的闭环。
- `Observation` 是下一步决策的事实输入。
- `State Machine` 用来约束 Agent 行为和停止条件。
- `Memory` 要保守，`Transcript` 要完整，`Context` 要精简。
- 写操作必须有权限、幂等、审计和验证。

## Week 1 认知升级问题

### Q1：第一周最重要的思想是什么？

学 Agent 先学边界。不要先问“模型能不能做”，要先问“这件事应该由模型、Runtime、工具、状态机、Workflow 还是人来负责”。

### Q2：为什么说 LLM 不是 Agent？

LLM 只负责根据输入生成输出，本身不天然拥有目标、工具、状态、权限、执行记录和停止条件。Agent 是把 LLM 放进 Runtime 后形成的目标执行系统。

### Q3：为什么 Runtime 比 Prompt 更接近工程核心？

Prompt 影响模型怎么回答；Runtime 决定模型能看到什么、能请求什么工具、动作能不能执行、结果如何记录、什么时候停止。这些才决定 Agent 能否落地。

### Q4：Workflow 和 Agent 的关系应该怎么理解？

Workflow 负责确定性控制，Agent 负责不确定性探索。生产系统里，关键业务状态应该交给 Workflow 或业务状态机，Agent 负责分析、辅助和生成建议。

### Q5：为什么 Observation 是从“聊天”走向“行动”的关键？

聊天只产生文本；行动会产生外部结果。Observation 把外部结果反馈给模型，让下一步基于事实，而不是基于模型自我想象。

### Q6：为什么状态机能提高 Agent 可靠性？

状态机把 Agent 行为限制在合法状态和合法转移内，防止无限循环、跳过权限、失败后继续执行、未验证就完成。

### Q7：Memory、Transcript、Context 最容易混淆在哪里？

很多人把所有历史都当 Memory。正确理解是：`Transcript` 记录完整事实，`Memory` 只保存长期稳定信息，`Context` 是本轮模型需要看到的精选信息。

### Q8：第一周最应该避免的错误认识是什么？

把 Agent 理解成“更会聊天的 LLM”。Agent 的核心不是更会说，而是能在 Runtime 控制下持续推进任务，并用证据证明状态变化。

### Q9：后端开发者学习 Agent 的优势是什么？

后端开发者熟悉状态机、权限、事务、幂等、日志、MQ、缓存和故障恢复，这些正是 Agent Runtime 工程化的核心能力。

### Q10：如果只记一句话，第一周应该记什么？

Agent = LLM 的推理能力 + Runtime 的执行控制 + Tool 的外部能力 + State / Trace 的工程约束。
