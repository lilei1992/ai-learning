# Week 07：AI Coding Agent 与企业内部 Agent QA

> 复习目标：理解 Coding Agent 如何在真实代码仓库中安全工作，并能设计企业内部 Agent MVP。

## Day 43：Coding Agent 基础

### Q1：Coding Agent 如何理解代码仓库？

先读规则文件和目录结构，再搜索相关代码，阅读调用链，结合测试和 diff 建立上下文。

### Q2：为什么 AGENTS.md / Rules 文件重要？

它们定义项目边界、编码规范、测试要求和禁止事项，是 Coding Agent 的工程约束。

### Q3：Coding Agent 的核心闭环是什么？

读规则 -> 找文件 -> 理解逻辑 -> 制定计划 -> 小范围修改 -> 运行验证 -> 汇报证据。

## Day 44：Task Decomposition

### Q1：代码任务为什么要拆解？

降低误改风险，让每一步都有证据和验证点。

### Q2：常见拆解方式是什么？

搜索入口、定位调用链、阅读数据结构、判断影响范围、修改、验证。

### Q3：什么时候应该停止拆解并问用户？

业务语义不明确、影响范围过大、需要改变公共契约或高风险状态逻辑时。

## Day 45：Context Isolation

### Q1：为什么需要上下文隔离？

避免多个子任务、多个文件、多个发现互相污染，导致模型混淆。

### Q2：Coding Agent 如何隔离上下文？

按子任务保留相关文件、关键证据和结论，汇总时只传摘要和引用。

### Q3：什么内容不该进入代码修改上下文？

无关文件、大量历史日志、未验证猜测、过期规则。

## Day 46：Result Aggregation

### Q1：Review Agent 如何汇总发现？

按严重级别、文件路径、问题描述、证据、风险、建议排序输出。

### Q2：为什么每条发现要带证据？

没有文件路径、行号或代码依据的建议不可验证，容易变成泛泛而谈。

### Q3：Review 输出应该避免什么？

避免风格偏好冒充缺陷，避免没有证据的风险判断。

## Day 47：Cost Control

### Q1：Coding Agent 成本主要来自哪里？

模型调用次数、上下文 token、长文件读取、重复搜索和高价模型使用。

### Q2：如何控制成本？

先搜索再读取，读取相关片段，缓存稳定规则，用摘要代替长历史，按任务选择模型。

### Q3：成本控制不能牺牲什么？

不能牺牲关键证据、权限校验和验证步骤。

## Day 48：Failure Handling

### Q1：测试失败后 Agent 应该怎么处理？

保留失败日志，判断是否与当前修改相关，必要时修复、回滚或报告未解决问题。

### Q2：权限不足时怎么办？

停止对应动作，说明缺少权限，给出用户可执行步骤或请求授权。

### Q3：语义不明确时为什么不能继续改？

继续改会把模型猜测变成代码事实，可能破坏业务规则。

## Day 49：Week 7 复盘

### Q1：Code Review Agent 的 MVP 怎么做？

只读 Git diff 和规则文件，输出风险点和建议，不直接修改代码。

### Q2：运维排障 Agent 的 MVP 怎么做？

只读日志、MySQL、Redis、ES，区分已验证事实和推测原因，高风险操作人工确认。

### Q3：本周最重要的一句话是什么？

AI Coding 的关键不是“生成代码”，而是正确选择上下文、小范围修改并用证据验证。

## Week 7 认知升级问题

### Q1：第七周最重要的思想是什么？

Multi-Agent 的价值来自受控分工，不来自“人多力量大”。没有分工、隔离和合并机制，多 Agent 只会放大混乱。

### Q2：什么时候适合使用 Multi-Agent？

任务可拆分、需要并行探索、需要多视角评审、单 Agent 上下文过大或需要专家角色时适合。

### Q3：什么时候不应该使用 Multi-Agent？

任务很小、状态强耦合、高风险写操作、结果需要强一致、无法定义子任务边界时不适合。

### Q4：Supervisor Agent 的核心职责是什么？

拆任务、分派、设置预算、隔离上下文、检查证据、合并结果、处理冲突和决定是否继续。

### Q5：Context Isolation 为什么重要？

如果所有子 Agent 共享全部上下文，会造成噪声扩散、错误传染、敏感信息扩大暴露和成本膨胀。

### Q6：结果聚合为什么不能简单拼接？

不同 Agent 的结论可能冲突、重复、证据不足或范围不同。聚合需要去重、排序、证据校验和冲突裁决。

### Q7：多 Agent 成本主要来自哪里？

来自多次模型调用、重复上下文、并行工具调用、结果合并、失败重试和更复杂的 Trace。

### Q8：第七周学懂的标志是什么？

你能判断一个任务是否值得多 Agent 化，并设计分工、上下文、预算、聚合和失败处理方案。

## Week 7 输出检查

- 能说明 Multi-Agent 的适用边界。
- 能设计 Supervisor / Worker 架构。
- 能写出任务拆解规则。
- 能设计子 Agent 上下文隔离策略。
- 能设计结果聚合格式。
- 能说明成本控制和失败隔离方案。
