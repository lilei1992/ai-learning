# Week 04：Context Engineering 与 Memory QA

> 复习目标：掌握 Agent 稳定性的关键：放什么上下文、不放什么上下文、记住什么、不记住什么。

## Day 22：Context Window

### Q1：Context Window 是什么？

模型一次调用能看到的 token 总量，包括指令、历史、工具结果、检索内容和输出预算。

### Q2：上下文窗口大就能随便塞吗？

不能。窗口变大只提高上限，不解决相关性、噪声、旧信息和敏感信息问题。

### Q3：上下文预算应该怎么分？

优先保留系统规则、用户目标、当前状态、关键证据和相关工具定义，再放历史摘要和检索内容。

## Day 23：Context Pollution

### Q1：什么是上下文污染？

无关、过期、错误、恶意或未验证信息进入上下文，影响模型判断。

### Q2：常见污染来源有哪些？

长日志、网页内容、旧 Observation、错误 Memory、无关文件、Prompt Injection。

### Q3：如何降低污染？

过滤、摘要、脱敏、来源标注、只放相关片段、区分事实和推测。

## Day 24：Tool Result Budget

### Q1：工具结果很长时怎么处理？

提取关键错误、摘要、证据、路径、行号和原始引用，不直接塞完整输出。

### Q2：什么内容必须保留？

能支撑下一步决策和最终结论的证据，例如失败测试名、错误摘要、文件路径、行号。

### Q3：raw output 应该怎么处理？

存入 Transcript 或外部存储，通过 `rawRef` 引用，需要时再读取。

## Day 25：Context Compaction

### Q1：为什么需要上下文压缩？

长任务会积累大量历史，必须压缩成关键决策、事实、未完成事项和风险。

### Q2：好的压缩摘要应包含什么？

目标、已完成动作、关键发现、当前状态、未解决问题、下一步建议。

### Q3：压缩的风险是什么？

丢失关键证据、混淆事实和推测、遗漏用户约束。

## Day 26：Prompt Cache

### Q1：什么适合缓存？

稳定系统指令、工具定义、项目规则、常用上下文和不频繁变化的文档摘要。

### Q2：缓存解决什么问题？

降低 token 成本、减少延迟、提升重复任务一致性。

### Q3：缓存的风险是什么？

使用过期规则、权限变化未生效、项目上下文不一致。

## Day 27：Memory Write Policy

### Q1：Working / Episodic / Semantic Memory 区别是什么？

Working Memory 是当前任务临时信息；Episodic Memory 是历史事件；Semantic Memory 是长期稳定知识。

### Q2：哪些信息可以写入长期 Memory？

用户稳定偏好、明确确认的项目规则、长期事实、可复用经验。

### Q3：哪些信息不应该写入 Memory？

一次性日志、未验证猜测、临时错误、敏感信息、过期状态。

## Day 28：Week 4 复盘

### Q1：Context Engineering 为什么比 Prompt Engineering 更重要？

Prompt 只规定怎么说，Context Engineering 决定模型能看到什么事实和证据。

### Q2：RAG 的关键链路是什么？

文档解析 -> chunk -> embedding -> 向量检索 -> 片段召回 -> 带引用回答。

### Q3：本周最重要的一句话是什么？

Agent 的质量很大程度取决于上下文选择质量，而不是只取决于提示词写得好不好。

## Week 4 核心思想追问

### Q1：为什么说 Context Window 是稀缺资源？

因为它同时承载规则、目标、状态、工具定义、工具结果、检索证据、Memory 和输出预算。窗口大只提高上限，不代表可以无筛选地放信息。

### Q2：上下文越多，模型一定越准确吗？

不一定。无关、过期、冲突或不可信内容会降低判断质量。Agent 需要的是高相关、高可信、可追溯的上下文，不是最长上下文。

### Q3：Runtime 在 Context Engineering 中负责什么？

负责选择、排序、压缩、脱敏、引用和记录上下文。模型负责基于上下文推理，但 Runtime 决定模型能看到什么。

### Q4：工具结果很长时，最小保留集是什么？

保留状态、摘要、关键证据、错误根因、路径、行号、记录 ID、rawRef。完整输出进入 Transcript 或 Artifact Store，不直接进入模型上下文。

### Q5：Context Compaction 和 Transcript 的区别是什么？

Transcript 是完整执行记录，追求可审计；Compaction 是继续执行所需的压缩视图，追求可用。压缩摘要不能替代完整记录。

### Q6：Memory 为什么必须有写入策略？

Memory 会影响未来任务。如果把临时日志、错误推测、敏感信息或过期状态写入长期记忆，会造成长期污染。

### Q7：RAG 为什么必须带引用来源？

引用来源让回答可验证、可追溯、可纠错。没有来源的回答很难判断是来自文档、模型常识还是幻觉。

### Q8：Prompt Cache 最大风险是什么？

使用过期上下文。例如旧工具 Schema、旧项目规则、旧权限信息继续影响模型行为。缓存必须绑定版本、hash 和失效策略。

### Q9：如何判断一段内容是否应该进入上下文？

看五个维度：是否相关、是否可信、是否新、是否必要、是否安全。只要不满足，就应该过滤、压缩或降级。

### Q10：上下文超限时，应该先裁剪什么？

通常先裁剪低相关背景、旧历史、重复日志、无关工具结果和低可信外部内容；保留系统规则、当前目标、当前状态、关键证据和输出预算。
