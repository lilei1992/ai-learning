## 4. 最小心智模型

可以先记住这句话：

> 工具失败、模型异常、上下文超限后如何恢复？ 这个问题的答案，最终都要落到“如何让 Agent 更可控、更准确、更可验证”。

从 Runtime 视角看，今天主题和下面链路有关：

```text
User Goal
-> Context / State
-> Model Decision
-> Runtime Control
-> Tool / Memory / Permission / Trace
-> Observation
-> Next Step
```

不要只问“模型会不会”，要问：

- Runtime 给模型看了什么？
- 模型输出如何被解析和校验？
- 工具或状态是否真的发生变化？
- 失败时有没有记录和恢复？
- 最终结论有没有证据？

## 5. 核心概念拆解

### 5.1 Retry（重试）

只适合安全、幂等、短暂失败的动作。

它解决的问题：处理网络抖动、临时限流、短暂文件锁这类可恢复错误。

工程落点：`RetryPolicy` 应定义最大次数、退避策略、可重试错误类型和幂等要求。

忽略后果：没有重试会降低成功率；无脑重试会放大副作用或造成成本失控。

### 5.2 Fallback（兜底）

主方案失败后切换到更安全或更简单方案。

它解决的问题：当首选工具或路径不可用时，Agent 可以降级完成部分目标。

工程落点：例如搜索工具失败后改用文件列表加局部读取，模型 A 超时后降级模型 B。

忽略后果：系统要么直接失败，要么模型编造替代结果而没有真实执行。

### 5.3 Ask User（询问用户）

业务语义、权限、风险不清时停止确认。

它解决的问题：在需求不清、权限不足、风险较高时，避免 Agent 擅自猜测或执行。

工程落点：`HumanInputRequest` 应包含问题、上下文、推荐选项、风险说明和恢复点。

忽略后果：Agent 可能做出用户不想要的修改，或者在高风险动作上越权。

### 5.4 Partial Result（部分结果）

无法完全完成时交付已验证发现和未完成原因。

它解决的问题：让失败也有价值，用户能知道完成了什么、卡在哪里、接下来怎么恢复。

工程落点：最终响应和 `TaskState` 都应记录 completed、pending、blockedReason、evidence。

忽略后果：用户只看到“失败了”，无法复用已完成工作，也无法判断下一步应该给什么信息。

## 6. 工程含义

今天主题的工程含义可以分成 5 层：

1. **边界**：明确模型、Runtime、工具、状态、用户各自负责什么。
2. **结构**：用接口、Schema、状态机、表结构或日志结构把能力固定下来。
3. **安全**：对高风险动作设置权限、审批、沙箱或只读限制。
4. **可恢复**：失败后能重试、降级、停止或交给用户处理。
5. **可验证**：最终结论必须能从工具结果、日志、状态或测试中找到证据。

## 11. 常见坑

- 只会解释概念，但说不出它在 Runtime 里的位置。
- 只相信模型输出，没有结构化校验。
- 没有考虑失败、超时、权限和审计。
- 把所有信息都塞进上下文，导致模型被噪声干扰。
- 没有最终验证，却在回答里声称任务完成。

## 13. 补充深度学习内容

### 13.1 Error Recovery 的目标不是“永远继续”

错误恢复不是让 Agent 不管遇到什么都继续执行。它的目标是：

> 在不扩大风险的前提下，尽量恢复任务；不能安全恢复时，停止并交付清晰证据。

Agent 的错误主要来自四类：

| 错误来源 | 示例 |
|----------|------|
| 模型错误 | 输出非法 JSON、选择不存在工具 |
| 工具错误 | 命令失败、文件不存在、HTTP 超时 |
| 上下文错误 | 上下文超限、旧信息污染 |
| 权限错误 | 高风险动作被拒绝、缺少访问权限 |

### 13.2 错误分类和策略

| 错误类型 | 是否重试 | 推荐处理 |
|----------|----------|----------|
| 参数错误 | 否 | 要求模型修正参数 |
| 权限拒绝 | 否 | 返回拒绝原因，必要时问用户 |
| 网络超时 | 可 | 限次数重试 |
| 只读查询失败 | 可 | 重试或换工具 |
| 写操作失败 | 谨慎 | 检查幂等和部分成功 |
| 上下文超限 | 否 | 压缩上下文 |
| 业务语义不明 | 否 | 停止并询问用户 |

### 13.3 Retry 必须依赖幂等

可以自动重试：

- 读取文件。
- 搜索代码。
- 查询只读数据库。
- 查询日志。
- 调用只读 HTTP API。

不能盲目重试：

- 写文件。
- 删除文件。
- 修改数据库。
- 发送邮件。
- 支付、退款、库存扣减。
- Git push / release。

原因很简单：非幂等操作重复执行会扩大副作用。

### 13.4 Recovery Policy 伪代码

```java
public RecoveryDecision decide(ToolError error, AgentAction action) {
    if (error.isPermissionDenied()) {
        return RecoveryDecision.askUserOrStop(error);
    }
    if (error.isValidationError()) {
        return RecoveryDecision.askModelToFix(error);
    }
    if (error.isTimeout() && action.isIdempotent()) {
        return RecoveryDecision.retryWithBackoff();
    }
    if (error.isContextTooLong()) {
        return RecoveryDecision.compactContext();
    }
    if (action.hasExternalSideEffect()) {
        return RecoveryDecision.stopForManualCheck();
    }
    return RecoveryDecision.fail(error);
}
```

### 13.5 Partial Success 要显式表达

假设写文件工具写到一半失败：

```text
Observation:
Tool partially succeeded.

Changed:
  src/UserService.java written.

Failed:
  src/UserServiceTest.java write failed due to permission denied.

Risk:
  Worktree may be inconsistent.

Next:
  Stop and ask user to inspect or grant permission.
```

不要把 partial success 当成 success。

### 13.6 最终回答中的错误透明度

错误恢复后最终回答要区分：

- 已完成。
- 已验证。
- 已部分完成。
- 未完成。
- 被权限阻塞。
- 需要用户确认。

一个好的失败回答：

```text
我没有继续修改，因为当前测试期望和业务状态流不一致。

已验证事实：
- 测试期望状态 ACTIVE。
- 实现默认状态 PENDING。
- 审核流程中才会设置 ACTIVE。

需要确认：
新用户创建后应该是 ACTIVE 还是 PENDING？
```
